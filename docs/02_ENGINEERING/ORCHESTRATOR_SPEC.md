# Спецификация оркестратора (DreamyDraw)

Технический контракт логики работы пайплайна. Описывает что делает оркестратор, в каком порядке, как обрабатывает циклы валидации и где может вмешаться пользователь.

Документ актуален для текущей реализации (стейтмашина на `current_node` в `Orchestrator`). После переезда на LangGraph документ будет переработан, но логика останется прежней.

---

## 1. Общий пайплайн

Оркестратор работает по принципу **Plan → Validate (with refinement loops) → Generate**.

Высокоуровневые фазы:

1. **Валидация входа** — проверка темы и совместимости с режимом.
2. **Планирование серии** — генерация пула идей и выбор финального плана.
3. **Контур валидации плана** — циклическая проверка плана с возможной редактурой и арбитражом пользователя.
4. **Генерация контента** — пакетная генерация текстов, согласование с пользователем, генерация иллюстраций.

Полный поток (упрощённо):

```
Тема + Конфигурация
        │
        ▼
[1] Safety Gate ──fail──▶ STOP
        │ pass
        ▼
[2] Config Match ──fail──▶ STOP
        │ pass
        ▼
[3] Series Planner (режимо-зависимый)
        │
        ▼
[4] Idea Scoring
        │
        ▼
[5] Score Normalize
        │
        ▼
[6] Idea Sampler ─────► план серии (N идей)
        │
        ▼
┌─────► [7] Plan Validator (режимо-зависимый)
│           │
│           ├─ APPROVED ─────────────────────┐
│           │                                │
│           └─ REJECTED                      │
│                │                           │
│                ▼                           │
│           [counter++]                      │
│                │                           │
│                ├─ counter < 3              │
│                │   │                       │
│                │   ▼                       │
│                │  [8] Plan Reviewer        │
│                │   │                       │
│                │   ▼                       │
│                │  [9] Plan Refiner ────────┘ (если REVISE)
│                │                           │
│                └─ counter >= 3             │
│                    │                       │
│                    ▼                       │
│                [USER ARBITRATION]          │
│                    │                       │
│                    ├─ "ок"/"хорошо" ───────│──▶ форсированное одобрение
│                    │                       │
│                    └─ комментарий ─────────│──▶ [8] → [9] → возврат в [7]
│                                            │
└────────────────────────────────────────────┘
                                             │
                                             ▼
                              [10] Text Generation (по каждой истории)
                                             │
                                             ▼
                              [11] User Confirmation (если режим check)
                                             │
                                             ▼
                              [12] Image Generation
                                             │
                                             ▼
                                          DONE
```

## 2. Стейтмашина

Реализована через поле `SessionState.current_node`. Список состояний и переходов.

| Состояние | Что происходит | Возможные следующие состояния |
|---|---|---|
| `start` | Стартовое состояние новой сессии | `safety_passed`, `failed` |
| `safety_passed` | Тема прошла проверку безопасности | `config_passed`, `failed` |
| `config_passed` | Тема совместима с режимом | `series_planned` |
| `series_planned` | План серии сгенерирован, готов к валидации | `plan_approved`, `plan_needs_refine`, `plan_needs_user_arbitration`, `failed` |
| `plan_needs_refine` | Валидатор отклонил план, идём в авторежим (счётчик < 3) | `series_planned` (после рефайна) |
| `plan_needs_user_arbitration` | Валидатор отклонил план в третий раз, требуется пользователь | `series_planned` (после ввода пользователя), `plan_approved` (форсированное одобрение) |
| `plan_approved` | План полностью одобрен валидатором | переход к генерации контента |
| `failed` | Терминальное состояние ошибки | — |

**Состояния фазы генерации контента** управляются не через `current_node`, а через флаги `is_completed` и `is_confirmed` у `StoryItem`. Это технический долг, см. раздел 8.

## 3. Описание узлов пайплайна

### Фаза I. Валидация входа

#### [1] Safety Gate
- **Идентификатор:** `_step_safety_gate`
- **Тип:** 🤖 LLM
- **Промпт:** `text/SAFETY_GATE.md`
- **Цель:** Отсечь темы с насилием, взрослым контентом, опасными сценариями.
- **Вход:** `topic`.
- **Выход:** обновлённое `current_node` (`safety_passed` или `failed`).
- **DoD:** Тема признана безопасной для детской аудитории.

#### [2] Config Match
- **Идентификатор:** `_step_config_match`
- **Тип:** 🤖 LLM + интерактив
- **Промпт:** `text/CONFIG_MATCH.md`
- **Цель:** Проверить совместимость темы и выбранного `truth_mode`.
- **Логика:** Если несовместимо — предложить пользователю смену режима через `input()`.
- **DoD:** Конфигурация признана логически верной.

### Фаза II. Планирование серии

#### [3] Series Planner
- **Идентификатор:** `_step_series_planner`
- **Тип:** 🤖 LLM
- **Промпт (режимо-зависимый):**
  - `text/planners/SERIES_PLANNER_TRUTH.md`
  - `text/planners/SERIES_PLANNER_MYTH.md`
  - `text/planners/SERIES_PLANNER_FAIRY_TALE.md`
- **Цель:** Сгенерировать пул из 10 идей в правильном жанровом регистре + `global_context` (описание персонажа).
- **Выход:** `session.ideas_pool` (10 объектов `Idea`), `session.global_context`.
- **DoD:** Пул из 10 валидных идей в нужном режиме.

#### [4] Idea Scoring
- **Идентификатор:** `_step_idea_scoring`
- **Тип:** 🤖 LLM
- **Промпт:** `text/IDEA_SCORING.md` (режимо-нейтральный, оценивает безопасность для возраста 3-5).
- **Цель:** Каждой идее присвоить `child_index` ∈ [0, 1].
- **Доп. логика:** идеи с `child_index < 0.3` отсеиваются. Если пул пуст — fallback "Прогулка в лесу".
- **DoD:** Все идеи имеют скор.

#### [5] Score Normalize
- **Идентификатор:** `_step_score_normalize`
- **Тип:** ⚙️ Детерминированный
- **Алгоритм:** Линейная нормализация со смещением `epsilon = 0.2` (см. раздел 7).
- **Цель:** Получить веса для взвешенной выборки.
- **DoD:** У каждой идеи `normalized_weight`, сумма весов = 1.

#### [6] Idea Sampler
- **Идентификатор:** `_step_idea_sampler`
- **Тип:** ⚙️ Детерминированный
- **Алгоритм:** Взвешенная случайная выборка без повторений (`random.choices` + удаление выбранной + ренормализация).
- **Цель:** Выбрать `count` уникальных идей из пула.
- **Выход:** `session.series_plan` (список названий тем), `session.full_plan_items` (список объектов `{theme, content}`).
- **DoD:** Финальный план из N идей готов к валидации.

### Фаза III. Контур валидации плана

#### [7] Plan Validator
- **Идентификатор:** `_step_plan_validator`
- **Тип:** 🤖 LLM
- **Промпт (режимо-зависимый):**
  - `text/validators/PLAN_VALIDATOR_TRUTH.md`
  - `text/validators/PLAN_VALIDATOR_MYTH.md`
  - `text/validators/PLAN_VALIDATOR_FAIRY_TALE.md`
- **Цель:** Проверить каждую тему на соответствие правилам режима ("красные флаги").
- **Логика:**
  - Темы из `approved_indices` пропускает (статус `ALREADY_APPROVED_BY_YOU`).
  - Для отклонённых — записывает `validator_feedback` с `invalid_indices`, `reasons`, `suggestions`.
  - При `APPROVED` всех тем — переход к `plan_approved`, **счётчик `validation_cycles` сбрасывается в 0**.
  - При `REJECTED` — **инкремент `validation_cycles += 1`**, переход к `plan_needs_refine` (если < 3) или `plan_needs_user_arbitration` (если ≥ 3).
- **DoD:** Все темы либо одобрены, либо помечены как требующие правки с указанием причины.

#### [8] Plan Reviewer (диспетчер)
- **Идентификатор:** часть `_step_plan_refine`
- **Тип:** 🤖 LLM
- **Промпт:** `text/PLAN_REVIEWER.md` (режимо-нейтральный).
- **Цель:** По каждой теме принять одно из решений:
  - `REVISE` — отправить на доработку редактору.
  - `KEEP_ORIGINAL` — оставить оригинал (только при явном требовании автора).
  - `ALREADY_OK` — тема не была отклонена и не требует изменений.
- **Доп. логика:**
  - При **пустом ответе** ревьюера — fallback `_build_fallback_decisions` (отклонённые → REVISE, остальные → ALREADY_OK).
  - **Страховка:** если ревьюер выбрал `KEEP_ORIGINAL` для отклонённой темы при пустом `user_comment` — принудительная замена на `REVISE`.
- **DoD:** Каждой теме присвоено решение.

#### [9] Plan Refiner
- **Идентификатор:** часть `_step_plan_refine`
- **Тип:** 🤖 LLM
- **Промпт (режимо-зависимый):**
  - `text/refiners/PLAN_REFINER_TRUTH.md`
  - `text/refiners/PLAN_REFINER_MYTH.md`
  - `text/refiners/PLAN_REFINER_FAIRY_TALE.md`
- **Цель:** Аккуратно довести `validator_suggestion` до финального вида (без отсебятины).
- **Принцип:** Редактор — НЕ автор. Берёт рекомендацию валидатора как основу, делает минимальные правки (опечатки, имя героя в серии). Запрещено добавлять новые предложения, эмоциональные обобщения, мораль.
- **Логика:** Запускается только для тем с решением `REVISE`. При пустом ответе → переход в `failed`.
- **DoD:** Все темы с решением `REVISE` обновлены в `full_plan_items`.

### Фаза IV. Генерация контента

#### [10] Text Generation
- **Идентификатор:** `_pipeline_content_generation` (часть до картинок)
- **Тип:** 🤖 LLM
- **Промпт:** `text/TEXT_BASE_PROMPT.md` + `truth_modes/{TRUTH|MYTH|FAIRY_TALE}.md` + `styles/{GENTLE|EDUCATIONAL|PLAYFUL}.md`.
- **Цель:** Сгенерировать полный текст истории + 2-3 вопроса по тексту.
- **Вход:** одобренный сюжет из `approved_plan_items[i]`.
- **Выход:** `story.text`, `story.questions`.
- **DoD:** Все истории имеют текст и вопросы.

#### [11] User Confirmation (только в режиме `check`)
- **Идентификатор:** в `main.py`
- **Тип:** Интерактив
- **Цель:** Пользователь подтверждает все тексты пакетно перед генерацией картинок.
- **Опции:** `y` (подтвердить), `n` (отмена), `r` (перегенерировать всё).
- **DoD:** Все `StoryItem.is_confirmed = True`.

#### [12] Image Generation
- **Идентификатор:** `_pipeline_content_generation` (часть после текстов)
- **Тип:** 🤖 Image Model
- **Промпт:** `image/IMAGE_BASE_PROMPT.md` + `image/styles/{CARTOON|WATERCOLOR|CLAY|NIGHT}.md`.
- **Цель:** Сгенерировать иллюстрацию для каждой истории.
- **Логика:** Синхронный вызов провайдера, сохранение в `output/<session_id>/story_{i}.png`.
- **DoD:** `session.is_completed = True`.

## 4. Циклы валидации и арбитраж пользователя

Ключевой механизм качества плана. Реализует контролируемый цикл "проверить → исправить → проверить" с защитой от бесконечной петли.

### Счётчик `validation_cycles`

- **Хранение:** `SessionState.validation_cycles: int`.
- **Семантика:** считает количество **REJECTED**-результатов от валидатора.
- **Инкремент:** в `_step_plan_validator` при статусе REJECTED.
- **Сброс в 0:** при полном одобрении плана (переход в `plan_approved`).
- **Назначение сброса:** валидатор будет переиспользоваться на других этапах (валидация текстов, картинок) — счётчик каждого этапа должен начинаться с нуля.

### Пороги и поведение

| `validation_cycles` после инкремента | Поведение |
|---|---|
| 1, 2 | Переход к `plan_needs_refine`. Авторежим: ревьюер и редактор работают самостоятельно, без участия пользователя. |
| 3 и выше | Переход к `plan_needs_user_arbitration`. Пользователь видит проблемные темы и может вмешаться. |
| > `MAX_VALIDATION_RETRIES` (5) | Принудительный переход в `failed`. Защита от бесконечной петли. |

### Арбитраж пользователя

Когда `validation_cycles >= 3`, в `main.py` показывается:
- Текущая версия проблемных тем (от редактора).
- Последнее замечание валидатора по каждой теме.

Пользователь может:
- Ввести **свободный комментарий** → передаётся ревьюеру/редактору как `user_feedback` для следующего цикла.
- Ввести **`ок` / `хорошо` / `хватит`** → форсированное одобрение текущей версии без валидации, счётчик сбрасывается.

### История правок

Все изменения тем сохраняются в `SessionState.revision_history: dict[str(idx), list[record]]`.
Каждая запись содержит:
- `source` — `planner`, `validator`, `reviewer:accept_suggestion`, `reviewer:keep_original`, `refiner`.
- `theme`, `content` — версия на момент записи.
- `note` — пояснение.

Используется в арбитраже для показа пользователю и для отладки (полная история сохраняется в JSON-сессии).

## 5. Режимная архитектура

Три режима правдивости (`TruthMode`): `Правда` / `Миф` / `Сказка`.

Промпты разделены по режимам в трёх ключевых точках пайплайна:

| Узел | Файлы |
|---|---|
| Series Planner | `planners/SERIES_PLANNER_{TRUTH,MYTH,FAIRY_TALE}.md` |
| Plan Validator | `validators/PLAN_VALIDATOR_{TRUTH,MYTH,FAIRY_TALE}.md` |
| Plan Refiner | `refiners/PLAN_REFINER_{TRUTH,MYTH,FAIRY_TALE}.md` |

**Режимо-нейтральные** промпты (один файл на все режимы):
- `SAFETY_GATE.md`, `CONFIG_MATCH.md`
- `IDEA_SCORING.md`
- `PLAN_REVIEWER.md`

**Текстовая генерация** использует другую модель подмешивания: один общий `TEXT_BASE_PROMPT.md` + один из `truth_modes/{TRUTH|MYTH|FAIRY_TALE}.md` + стиль текста.

Маппинг значения режима → суффикс файла реализован в `PromptBuilder._map_truth_mode_to_suffix`:
- `Правда` → `TRUTH`
- `Миф` → `MYTH`
- `Сказка` → `FAIRY_TALE`

## 6. Состояние сессии

Хранится в `SessionState` (см. `src/models/schemas.py`).

### Ключевые поля для оркестрации

| Поле | Тип | Назначение |
|---|---|---|
| `current_node` | str | Текущий узел стейтмашины |
| `series_plan` | List[str] | Список названий тем |
| `full_plan_items` | List[dict] | Полный план с `theme` и `content` |
| `approved_plan_items` | dict | Чистовик одобренных тем по индексу |
| `approved_indices` | List[int] | Индексы тем, прошедших валидацию |
| `global_context` | str | Описание персонажа и мира серии |
| `ideas_pool` | List[Idea] | Пул идей до сэмплинга |
| `validation_cycles` | int | Счётчик REJECTED-циклов |
| `validator_feedback` | str (JSON) | Последний фидбек валидатора |
| `user_feedback` | Optional[str] | Комментарий пользователя для рефайна |
| `revision_history` | dict | История всех правок по темам |
| `stories` | List[StoryItem] | Готовые истории |
| `is_completed` | bool | Флаг завершения всей генерации |

### Сериализация

Сессия полностью сериализуется в JSON через `JSONStorage`. Сохранение — после каждого изменения состояния. Это даёт возможность восстановления сессии (`--session <id>`).

## 7. Конфигурация и константы

Текущие константы (объявлены в `orchestrator.py`, после рефакторинга должны переехать в `settings.py` или `constants.py`):

| Константа | Значение | Назначение |
|---|---|---|
| `USER_ARBITRATION_THRESHOLD` | 3 | Порог REJECTED-циклов до подключения пользователя |
| `MAX_VALIDATION_RETRIES` | 5 | Абсолютный лимит циклов (защита от петли) |

Локальные магические числа в коде (требуют переезда в конфиг):

| Где | Значение | Что делает |
|---|---|---|
| `_step_idea_scoring` | `0.3` | Минимальный `child_index` для прохождения фильтра |
| `_step_score_normalize` | `epsilon = 0.2` | Смещение для линейной нормализации (живучесть слабых идей) |
| `_step_series_planner` | `10` | Размер пула идей |
| `IDEA_SCORING.md` | `0.5` | Fallback-значение скора при ошибке |

## 8. Технический долг и точки расширения

### Известный технический долг

1. **God object Orchestrator.** Класс совмещает логику валидации, планирования, скоринга, нормализации, рефайна, генерации текста, генерации картинок. По-хорошему — 5-7 отдельных классов.
2. **Стейтмашина на строках.** `current_node` — строка, переходы захардкожены в `if/elif`. Нет явной таблицы переходов. Запланирована миграция на LangGraph.
3. **Магические числа в коде.** Не вынесены в конфиг (запланирован рефакторинг).
4. **`print()` вместо `logging`.** Запланирован переход на стандартный logger.
5. **Парсинг JSON через копипасту.** `response_raw.replace("```json", "").replace("```", "").strip()` встречается ~10 раз.
6. **Изменяемые дефолты в Pydantic-моделях** (`dict = {}`, `List[dict] = []`). Не критично в pydantic v2, но архитектурно неаккуратно.
7. **`StoryItem.retry_count`** оставлен в схеме, но больше не используется как раньше (заменён на `validation_cycles` уровня сессии).
8. **Отсутствие тестов.** Особенно критично для стейтмашины и циклов валидации.

### Запланированные расширения

1. **Валидация и редактура текстов** (после генерации, перед согласованием с пользователем). Будет использовать тот же паттерн `validator → reviewer → refiner`, что и для плана. Потребует новых промптов: `TEXT_VALIDATOR_*.md`, `TEXT_REFINER_*.md`. Логика арбитража и счётчик переиспользуются.
2. **Переход на LangGraph + Langfuse.** Стейтмашина станет явным графом, появится observability вызовов LLM.
3. **Асинхронная генерация картинок.** `asyncio.gather` для пакетной отрисовки. Делается после стабилизации текущей логики.
4. **JSON-база знаний** для подмешивания фактов о популярных темах (лягушки, ёжики, белки и т.п.) в промпты валидатора и редактора. Альтернатива классическому RAG для предсказуемой предметной области.
5. **Видео-провайдер** рядом с image-провайдером.
6. **Загрузка фото ребёнка** как референса персонажа. Большая отдельная фича с юридическими и техническими нюансами.

### Точки расширения для LangGraph

Каждый `_step_*` метод оркестратора **может стать узлом графа** с минимальными изменениями. Сигнатура `(SessionState) -> SessionState` уже соответствует ожидаемой LangGraph-нодой. Главные изменения при миграции:
- Явное описание рёбер (переходов) вместо `if/elif` в `run_pipeline`.
- Обёртка вокруг циклических узлов (валидатор↔рефайнер).
- Точки выхода для интерактива с пользователем (`plan_needs_user_arbitration`, `User Confirmation`) станут специальными "interrupt" узлами.

## 9. Соглашения и инварианты

Список инвариантов, которые должны соблюдаться в любой момент:

1. **`approved_indices` ⊆ ключи `approved_plan_items`** — если индекс одобрен, для него есть данные в чистовике.
2. **`validation_cycles >= 0`** всегда. Сброс в 0 — только при `plan_approved`.
3. **`validation_cycles <= MAX_VALIDATION_RETRIES`** иначе сессия идёт в `failed`.
4. **`session.user_feedback` обнуляется** после использования (в конце `_step_plan_refine`).
5. **`current_node == "failed"`** — терминальное состояние, из него нет переходов.
6. **`session.is_completed == True`** — флаг ставится только после успешной генерации картинок.
7. **`session.full_plan_items[i]` всегда актуален** для всех `i in range(len(series_plan))`.
