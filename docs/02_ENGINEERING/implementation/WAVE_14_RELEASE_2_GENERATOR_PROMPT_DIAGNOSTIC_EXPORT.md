# Wave 14 Task — Release 2 generator prompt diagnostic export

Status: **planned; diagnostics only; implementation not started**.  
Owner: **Engineering Agent**.  
Release: **Release 2**.

## Коротко

Нужно создать временную директорию с тремя Markdown-кейсами:

1. обычная сказочная лиса;
2. сказочная лиса в русской народной манере;
3. сказочная лиса с `CHUKOVSKY_STYLE`.

Для каждого кейса показать:

- параметры запуска;
- normalized request;
- resolved layers;
- состав prompt по частям;
- источник и назначение каждой части;
- полный текст каждой части;
- точный итоговый prompt в том виде, в котором он передаётся генератору.

После подготовки не анализировать и не исправлять содержимое prompt.

---

## 1. Цель

Подготовить воспроизводимые диагностические материалы для анализа текущего prompt, который получает `candidate_text_generator`.

Артефакты будут переданы prompt engineers, которые отдельно разберут:

- перегруженность;
- повторения;
- конфликты;
- порядок частей;
- технический шум;
- корректность разделения ответственности.

После анализа будет принято отдельное решение, кто и какие изменения реализует.

## 2. Временная директория

Создать директорию примерно такой структуры:

```text
docs/02_ENGINEERING/implementation/_TEMP_PROMPT_DIAGNOSTICS/
├── README.md
├── 01_FAIRY_FOX_BASE.md
├── 02_FAIRY_FOX_RUSSIAN_FOLK.md
└── 03_FAIRY_FOX_CHUKOVSKY.md
```

Название можно немного скорректировать, но временный статус должен быть очевиден.

В `README.md` указать:

- назначение директории;
- способ воспроизведения артефактов;
- что файлы не являются prompt assets или source of truth;
- что `PromptRegistry` не должен их загружать;
- что директория удаляется после завершения анализа и реализации prompt redesign;
- что удаление входит в cleanup следующей соответствующей волны.

## 3. Общие controlled parameters

Для всех трёх кейсов использовать одинаковые параметры:

```text
output_count = 2
target_age = 3
truth_mode = FAIRY_TALE
cultural_context = RUSSIAN_FOLK
utility_mode = NARRATIVE
```

`PromptRegistry` должен загружаться из:

```text
prompts/cultural_contexts/russian_folk/
```

Другие параметры менять только тогда, когда это необходимо для активации исследуемого style/reference layer.

## 4. Диагностические кейсы

### 4.1 Кейс 1 — базовая сказочная лиса

Пример `raw_text`:

```text
Короткие истории про лисичку.
```

Ожидается:

- `FAIRY_TALE`;
- fox entity layer;
- без явно выбранного `RUSSIAN_FOLK_TALE`;
- без `CHUKOVSKY_STYLE`.

Этот кейс показывает фактическое поведение сказочной лисы внутри русского cultural root без дополнительного style label.

### 4.2 Кейс 2 — русская народная манера

Пример `raw_text`:

```text
Короткие истории про лисичку в русской народной манере.
```

Ожидается подключение:

```text
RUSSIAN_FOLK_TALE
```

Остальные controlled parameters должны совпадать с первым кейсом.

### 4.3 Кейс 3 — Чуковский

Пример `raw_text`:

```text
Короткие истории про лисичку в стиле Чуковского.
```

Ожидается подключение:

```text
CHUKOVSKY_STYLE
```

Не добавлять `RUSSIAN_FOLK_TALE` вручную, если текущий runtime сам его не выбирает. Артефакт должен отражать фактическую сборку.

## 5. Структура каждого Markdown-файла

### A. Входные данные

Указать:

- `raw_text`;
- controlled parameters;
- model name, если он входит в prompt;
- candidate count;
- cultural prompt root.

### B. Normalized request

Привести фактический normalized request или его generator-facing summary.

### C. Resolved prompt context

Указать по порядку:

- `ordered_layer_ids`;
- fallback layers;
- unresolved details;
- hard details;
- soft preferences.

### D. Карта составных частей

Сделать компактную таблицу:

| № | Часть | Источник | Назначение | Размер |
|---|---|---|---|---:|

В карте должны быть отражены как минимум:

- executor wrapper;
- generator task и Python suffixes;
- normalized request summary;
- length policy;
- stage inputs;
- prompt context;
- bodies всех активных layers;
- stage instructions/context blocks;
- hard/soft/unresolved details;
- required JSON output shape.

Для каждого layer body отдельно указать:

- layer ID;
- файл-источник;
- почему он выбран;
- размер;
- кратко, о чём его содержание.

### E. Полные части prompt

После карты привести полный текст каждой существенной части без сокращений.

### F. Точный итоговый prompt

В конце файла привести полный prompt именно в том виде, в котором он передаётся LLM executor.

Не пересобирать его вручную. Он должен быть получен из фактического runtime-пути.

Добавить:

- общий размер;
- стабильный hash;
- ordered layer IDs;
- дату/версию диагностики.

### G. Замечания prompt engineers

Оставить пустой раздел:

```markdown
## Замечания prompt engineers

_Заполняется на следующем этапе._
```

## 6. Требования к точности

- Использовать активный путь `scripts/run_stage1_2_mvp.py` и текущие Stage 1/composer/executor компоненты.
- Не вызывать внешний LLM.
- Не включать API keys, environment secrets или provider responses.
- Не редактировать prompt dump вручную.
- Зафиксировать способ проверки, что Markdown содержит тот же prompt, который построил executor.
- Учесть вложенные Markdown code fences внутри layer bodies: внешний fence должен быть достаточно длинным.
- Не подменять отсутствующие layers ожидаемым целевым поведением.

## 7. Допустимая техническая реализация

Можно использовать:

- существующий LLM debug/capture механизм;
- тестовый capture provider без внешнего вызова;
- небольшой временный export script, если без него артефакты нельзя воспроизводимо обновлять.

Если создаётся временный скрипт, его расположение и команда запуска должны быть описаны в `README.md`. Он также должен быть удалён вместе с диагностической директорией после завершения работы.

## 8. Что не входит в задачу

Не нужно:

- анализировать качество prompt;
- исправлять prompt bodies;
- менять layer resolution;
- менять composer;
- добавлять cultural prompt layer;
- автоматически подключать `RUSSIAN_FOLK_TALE`;
- сокращать payload;
- вызывать реального LLM;
- генерировать или оценивать тексты;
- изменять Stage 2 pipeline;
- создавать постоянную диагностическую инфраструктуру.

## 9. Проверка результата

Перед передачей убедиться, что:

- созданы все три Markdown-кейса;
- controlled parameters одинаковы;
- различия resolved layers соответствуют фактическому runtime;
- каждый файл содержит карту частей и полный итоговый prompt;
- итоговый prompt получен автоматически;
- hashes и размеры указаны;
- внешний provider не вызывался;
- существующие автоматические тесты продолжают проходить;
- вне временной директории нет непредусмотренных изменений.

## 10. Результат работы

После выполнения прислать короткий отчёт:

- созданные файлы;
- команда воспроизведения;
- размеры трёх итоговых prompts;
- ordered layer IDs для каждого кейса;
- подтверждение отсутствия внешних LLM-вызовов;
- результат тестового прогона;
- любые технические ограничения диагностического формата.
