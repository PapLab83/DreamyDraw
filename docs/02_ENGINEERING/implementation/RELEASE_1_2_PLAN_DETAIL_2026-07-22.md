# Детализация плана Release 1–2 от 2026-07-22

## 1. Культурный контекст

Текущий `RUSSIAN_FOLK` предлагается переименовать в:

```text
RUSSIAN
```

Причина: cultural context обозначает общую русскую культурную территорию, внутри которой доступны разные сказочные стили:

- русская народная сказка;
- Чуковский;
- Пушкин;
- Маршак;
- будущие русские стили.

`russian_folk` при этом остаётся отдельным стилем внутри `RUSSIAN`, а не названием всего культурного контекста.

## 2. Первый набор персонажей

Сначала технология отрабатывается на трёх персонажах в русском народном стиле:

- лиса;
- волк;
- заяц.

Для каждого подготавливается отдельный цельный prompt с характером персонажа, фольклорными мотивами, примерами и требованиями к результату.

## 3. Fallback для неизвестного персонажа

Если пользователь просит персонажа, для которого нет отдельного prompt в выбранном стиле:

- персонаж определяется из запроса;
- его название сохраняется в normalized state;
- сборщик использует общий `SINGLE_CHARACTER` prompt выбранного стиля.

Например, для мыши без `MOUSE.md` будет использован:

```text
russian_folk/fallbacks/SINGLE_CHARACTER.md
```

## 4. Несколько персонажей

Если пользователь просит историю с несколькими персонажами, используется отдельный общий prompt:

```text
fallbacks/MULTI_CHARACTER.md
```

Он содержит правила и примеры построения сказок с несколькими героями.

## 5. Reference styles

Для Чуковского, Пушкина и Маршака на первом этапе достаточно fallback-prompts:

- `SINGLE_CHARACTER.md`;
- `MULTI_CHARACTER.md`.

Директории `characters/` создаются сразу как архитектурные точки расширения, но могут пока оставаться без character-specific prompts.

## 7. Порядок сборки

```text
cultural_context = RUSSIAN
→ truth_mode = FAIRY_TALE
→ выбор стиля
→ определение одного или нескольких персонажей
→ поиск character-specific prompt
→ SINGLE_CHARACTER или MULTI_CHARACTER fallback при отсутствии подходящего prompt
→ добавление runtime-параметров
→ передача цельного prompt генератору
```

## 8. Определение персонажей

Для определения персонажей используется узкий AI-агент с одной задачей:

- определить главного персонажа;
- перечислить остальных персонажей;
- определить, один персонаж или несколько.

Выбор файлов и fallback выполняет уже детерминированный сборщик.

## 9. Безопасный рефакторинг

- Сначала готовятся и проверяются новые prompts.
- Текущая сборка продолжает работать без изменений.
- Новая логика подключается параллельно.
- Маршрутизация проверяется на fox, wolf, hare, unknown single и multi-character cases.
- После проверки новая сборка заменяет текущую.
- Старые layers и временные ветки удаляются отдельным cleanup-этапом.
