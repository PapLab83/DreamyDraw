# TARGET_ORCHESTRATION_LOGIC.md

# Целевая логика оркестрации

## 1. Назначение документа

Документ описывает целевую логику работы оркестратора с точки зрения пользовательского сценария, подготовки параметров генерации, принятия решений системой, взаимодействия с пользователем и последовательной подготовки итогового результата.

Документ фиксирует не техническую реализацию LangGraph-графа, а логическую модель продукта:

* что пользователь вводит;
* как система понимает и нормализует запрос;
* какие параметры должна собрать перед запуском генерации;
* когда система может продолжать автоматически;
* когда должна уточнить запрос;
* как обрабатываются неполные, пустые, бессмысленные или противоречивые запросы;
* как формируется и валидируется набор параметров для дальнейшей генерации;
* как пользователь видит итоговую интерпретацию своего запроса перед запуском основного пайплайна;
* как после фиксации параметров система подбирает prompt layers / `.md`-файлы;
* как дальше организуется основной процесс подготовки результата: планирование, валидация, редактура, генерация текстов, проверка текстов, генерация иллюстраций и финальная выдача.

На текущем этапе документа подробно описан первый крупный блок — сбор и нормализация параметров генерации. Остальные этапы основного пайплайна пока обозначаются верхнеуровнево и будут детализироваться в следующих итерациях документа.

Техническая реализация — конкретные ноды LangGraph, routing-функции, структура `SessionState`, interrupt/resume и Langfuse-наблюдаемость — должна описываться отдельно в технической спецификации. Этот документ является логическим основанием для такой реализации.

---

## 2. Общий принцип целевой оркестрации

На текущем уровне детализации в целевой оркестрации предварительно выделяются два крупных блока:

1. **Сбор, нормализация и валидация параметров генерации**
   Система получает произвольный пользовательский ввод и превращает его в внятный, полный, непротиворечивый и исполнимый набор параметров.

2. **Основной пайплайн подготовки результата**
   После фиксации параметров система подбирает соответствующие prompt layers / `.md`-файлы, формирует план серии, валидирует и улучшает его, генерирует тексты, проверяет и редактирует их, готовит иллюстрации и отдаёт пользователю финальный результат.

Это деление не является окончательным. По мере детализации документа количество крупных этапов может измениться: некоторые блоки могут быть выделены отдельно, объединены или разложены на несколько самостоятельных логических контуров.

Ключевой принцип: основной пайплайн генерации не должен запускаться, пока система не получила минимально необходимый набор базовых параметров и не признала его достаточно понятным, согласованным и исполнимым.

## 3. Верхнеуровневая схема целевой оркестрации

Раздел содержит две схемы:

1. короткую обзорную схему — для быстрого понимания общего маршрута запроса;
2. детальную схему первого блока — для фиксации логики анализа, уточнений, повторной классификации и валидации параметров.

Такое разделение нужно потому, что одна схема не может одновременно быть короткой и достаточно точной. Обзорная схема показывает основную траекторию, а детальная раскрывает развилки и правила первого блока.

---

### 3.1 Короткая обзорная схема

```text
[0] Пользовательский ввод
        │
        ▼
[1] Анализ и нормализация запроса
        │
        ▼
[2] Уточняющий контур, если нужен
        │
        ▼
[3] Candidate layer resolution
        │
        ▼
[4] Финальная валидация параметров
        │
        ▼
[5] Preview исполнимой интерпретации
        │
        ▼
[6] Подготовка prompt execution context
        │
        ▼
[7] Основной pipeline подготовки результата
```

Ключевая идея обзорной схемы:

* сначала система понимает запрос и собирает параметры;
* если данных не хватает или есть конфликт, запускается уточняющий контур;
* до preview система должна уже проверить, что выбранная интерпретация исполнима через доступные режимы и prompt layers;
* preview показывает пользователю не предварительную догадку, а уже исполнимую интерпретацию;
* после preview система фиксирует prompt execution context для основного pipeline: выбранные layer ids, fallback-решения, ограничения и общий контекст исполнения.

---

### 3.2 Детальная схема первого блока: анализ, уточнение и валидация параметров

```text
[0] Пользовательский ввод
        │
        ▼
[1] Анализ и нормализация ввода
        │
        ├─ извлечение возможных параметров
        ├─ оценка полноты базовых параметров
        ├─ оценка уверенности по базовым параметрам
        ├─ выделение hard_details / soft_preferences
        ├─ prompt-aware metadata lookup
        │     ├─ чтение metadata доступных режимов / стилей / подстилей
        │     ├─ сопоставление пользовательских формулировок с поддерживаемыми слоями
        │     └─ предварительная fallback-оценка
        └─ обновление нормализованного объекта параметров
        │
        ▼
[2] Классификация состояния параметров
        │
        ├─ полный и понятный запрос
        │       │
        │       ▼
        │   [4] Candidate layer resolution
        │
        ├─ неполный или неоднозначный запрос
        │       │
        │       ▼
        │   [3] Уточняющий контур: пакетное уточнение
        │
        ├─ пустой или бессмысленный запрос
        │       │
        │       ▼
        │   [3] Уточняющий контур: объяснение продукта + стартовые варианты
        │
        └─ противоречивый запрос
                │
                ▼
            [3] Уточняющий контур: объяснение противоречия + альтернативы

[3] Уточняющий контур
        │
        ├─ пользователь выбрал готовый вариант
        ├─ пользователь уточнил свободным текстом
        ├─ пользователь выбрал непротиворечивую альтернативу
        ├─ достигнут лимит + есть внятная интерпретация
        │       │
        │       ▼
        │   [1] Повторный анализ выбранного / обновлённого ввода
        │
        └─ достигнут лимит + нет внятной темы или выбранного варианта
                │
                ▼
            STOP / ожидание нового ввода

[1] Повторный анализ выбранного / обновлённого ввода
        │
        ▼
[2] Повторная классификация состояния параметров
        │
        ├─ полный и понятный запрос ───────────────▶ [4]
        ├─ всё ещё неполный / неоднозначный ───────▶ [3]
        ├─ снова пустой / бессмысленный ───────────▶ [3] или STOP
        └─ появилось новое противоречие ───────────▶ [3]

[4] Candidate layer resolution
        │
        ├─ проверка, что для базовых параметров есть поддерживаемые режимы / слои
        ├─ определение candidate prompt layers по metadata
        ├─ фиксация ближайших поддерживаемых слоёв, если точного слоя нет
        ├─ фиксация fallback-решений
        └─ проверка, что preview не обещает неподдерживаемую стилизацию
        │
        ▼
[5] Финальная валидация нормализованного объекта
        │
        ├─ pass
        │     │
        │     ▼
        │ [6] Preview исполнимой интерпретации
        │     │
        │     ▼
        │ [7] Подготовка prompt execution context
        │     │
        │     ├─ фиксация выбранных layer ids
        │     ├─ фиксация fallback-решений
        │     ├─ подготовка общего контекста параметров
        │     ├─ подготовка hard constraints / soft preferences
        │     └─ передача prompt context в основной pipeline
        │     │
        │     ▼
        │ [8] Основной pipeline подготовки результата
        │
        └─ fail
              │
              ├─ повторная классификация причины
              ├─ возврат в уточняющий контур
              ├─ предложение альтернатив
              └─ STOP, если запрос нельзя привести к исполнимому виду
```

Основной pipeline подготовки результата на текущем этапе документа обозначается верхнеуровнево:

```text
[8] Основной pipeline подготовки результата
        │
        ├─ генерация пула candidate texts
        ├─ проверка разнообразия тем
        ├─ hard gates
        ├─ scoring / ranking
        ├─ последовательная валидация и редактура
        ├─ набор approved texts
        ├─ fallback при нехватке approved texts
        └─ выход второго этапа для генерации иллюстраций
```

Подробная логика этих этапов описана в разделе 5.

### 3.3 Пояснения к схемам

#### 3.3.1 Анализ и финальная валидация — разные процессы

Анализ и нормализация отвечают на вопрос:

```text
Что пользователь имел в виду и какие параметры можно извлечь из его ввода?
```

Финальная валидация отвечает на другой вопрос:

```text
Можно ли с этим нормализованным объектом запускать основной pipeline?
```

Поэтому после уточнений система не должна сразу переходить к финальной валидации. Сначала она повторно анализирует обновлённый ввод или выбранный вариант, затем повторно классифицирует состояние параметров. Только если параметры стали полными и понятными, система переходит к candidate layer resolution и финальной валидации.

#### 3.3.2 Уточняющий контур общий для разных типов проблем

Уточняющий контур может запускаться не только для неполного или неоднозначного запроса.

Он также используется, если:

* пользовательский ввод пустой или бессмысленный;
* запрос содержит противоречие;
* пользователь указал неподдерживаемый стиль как жёсткое требование;
* hard details конфликтуют с выбранным режимом;
* система не может собрать исполнимый набор параметров без дополнительного выбора пользователя.

Лимит уточнений относится ко всему уточняющему контуру, а не к одной конкретной ветке.

#### 3.3.3 Поведение после лимита уточнений

Базовое поведение остаётся осторожным:

* если после лимита уточнений есть внятная интерпретация, система может передать её на финальную валидацию;
* если после лимита уточнений нет внятной темы, выбранного варианта или понятной интерпретации, основной pipeline не запускается.

Для пустого или бессмысленного ввода это правило особенно важно: система не должна запускать генерацию полностью на собственных догадках, если пользователь так и не выбрал вариант и не дал понятную тему.

В будущем может быть добавлен отдельный режим `best-effort generation`: система после лимита уточнений сама выбирает безопасные дефолты, предупреждает пользователя и запускает генерацию. Это отдельное продуктовое решение. Его плюс — меньше тупиков для пользователя. Его минус — риск нерелевантного результата и лишних затрат на генерацию.

#### 3.3.4 Prompt layers используются в двух разных режимах

Prompt layers участвуют в логике оркестрации дважды, но в разных ролях.

Первый режим — **prompt-aware metadata lookup** на этапе анализа и уточнения запроса.

На этом этапе система использует metadata prompt layers, чтобы:

* понять, какие режимы, стили и подстили доступны;
* сопоставить пользовательские формулировки с поддерживаемыми слоями;
* не предлагать пользователю неподдерживаемую стилизацию как гарантированную;
* определить возможные fallback-варианты;
* подготовить человеко-понятное объяснение для уточнения или preview.

Второй режим — **подготовка prompt execution context** после preview.

На этом этапе система уже не решает, что обещать пользователю. Эта интерпретация должна быть согласована до preview. После preview система фиксирует выбранные layer ids, fallback-решения, общий контекст параметров, hard constraints и soft preferences, чтобы передать этот набор в основной pipeline.

Фактическая загрузка и сборка `.md` prompt bodies происходит внутри основного pipeline по мере необходимости конкретного агента или этапа. Например, planner собирает свои prompt layers, validator — свои, text generator — свои, image generator — свои.

Иными словами:

```text
до preview:
metadata lookup + candidate layer resolution
        → система понимает, что она может корректно обещать пользователю

после preview:
prompt execution context preparation
        → система фиксирует выбранные layer ids, fallback-решения и общий контекст исполнения

внутри основного pipeline:
stage-specific prompt assembly
        → конкретные агенты загружают и собирают нужные им `.md`-слои
```

Preview должен показывать уже исполнимую интерпретацию, а не предварительную догадку, которая может измениться после загрузки слоёв.

## 4. Сбор и нормализация параметров генерации

### 4.1 Цель этапа

Цель этапа — превратить произвольный пользовательский ввод в минимально достаточный, нормализованный и согласованный набор параметров, на основе которого можно запускать основной пайплайн генерации.

Пользователь не обязан заполнять форму или заранее знать все режимы системы. Он может написать запрос свободным текстом, неполно, нестрого или даже оставить поле пустым. Задача оркестратора — помочь пользователю прийти к исполнимому запросу без лишнего трения.

Пример полного запроса:

```text
Сделай 5 коротких натуралистичных историй про ёжика зимой в лесу для ребёнка 3 лет, в спокойном повествовательном стиле.
```

Пример неполного запроса:

```text
Сделай про ёжика.
```

Пример противоречивого запроса:

```text
Расскажи правдивую поучительную историю про ёжика на ковре-самолёте.
```

Пример пустого или бессмысленного ввода:

```text
...
```

или

```text
ываыва 123
```

Во всех случаях система должна прийти к одному из двух результатов:

1. сформирован полный и согласованный набор параметров;
2. пользователь отказался продолжать или не выбрал ни один из предложенных вариантов.

---

### 4.2 Базовые обязательные параметры

Перед запуском основного пайплайна должны быть заполнены базовые параметры генерации.

На старте целевой логики базовыми считаются:

| Параметр            | Назначение                                        | Пример                                                            |
| ------------------- | ------------------------------------------------- | ----------------------------------------------------------------- |
| `content_format`    | Формат результата                                 | `story`                                                           |
| `truth_mode`        | Режим отношения к реальности                      | `Правда`, `Миф`, `Сказка`                                         |
| `utility_mode`      | Цель генерации / назначение результата            | `Повествовательный`, `Поучительный`, `Английский`                 |
| `target_age`        | Возраст ребёнка                                   | `3`, `3.5`, `4`, `4.5`, `5`                                       |
| `main_subject`      | Главный персонаж, объект или тема                 | `ёжик`, `маленький Мук`, `белка зимой`                            |
| `output_count`      | Количество историй / элементов результата         | `3`, `5`, `7`                                                     |
| `audience_language` | Язык пользовательского взаимодействия             | `ru`                                                              |
| `result_language`   | Язык итогового результата                         | `ru`, в будущем — `en` и другие                                   |
| `text_style_base`   | Базовый тон / стиль текста                        | `спокойный`, `познавательный`, `игровой`                          |
| `substyle`          | Более конкретный подстиль или жанровая стилизация | `натуралистичный`, `русско-народный`, `в духе Чуковского`, `null` |

Часть параметров имеет ограниченный набор допустимых значений. Например, `content_format`, `truth_mode`, `utility_mode`, `target_age`, `audience_language` и `result_language` должны быть выбраны из заранее известных значений системы.

Другие параметры могут быть более свободными, но всё равно должны быть нормализованы. Например, пользователь может написать «как у Чуковского», а система должна интерпретировать это как конкретный поддерживаемый подстиль, если такой prompt layer существует.

`substyle` является базовым полем нормализованного объекта, но не всегда обязательным значением. Оно может быть `null`, если пользователь не указал конкретный подстиль, если подстиль не нужен для выбранного формата результата или если точный подстиль не найден в prompt-базе. В таком случае система использует базовые слои стиля и сохраняет пользовательские пожелания в `hard_details` или `soft_preferences`, если они важны для дальнейшей генерации.

На старте продукт ориентирован на русскоязычную аудиторию:

* `audience_language = ru`;
* уточнения, preview и объяснения показываются по-русски;
* `utility_mode = ENGLISH` не означает автоматически, что весь интерфейс или весь результат должен быть на английском;
* `utility_mode = ENGLISH` означает образовательную цель, связанную с изучением английского;
* язык итогового результата должен развиваться отдельно через `result_language` или аналогичное поле.

Поле `content_format` фиксирует формат результата. На старте основным форматом считается:

```text
content_format = story
```

В будущем рядом могут появиться другие форматы, например:

* `picture_riddle`;
* `english_card`;
* `storyboard`;
* `dialogue_practice`.

Добавление новых форматов не должно требовать пересборки всей логики сбора параметров. Формат результата должен становиться одним из базовых параметров, который влияет на дальнейший подбор prompt layers и основной pipeline.

---

### 4.3 Типы параметров

Параметры генерации делятся на три группы.

#### 4.3.1 Базовые обязательные параметры

Это параметры, без которых нельзя запускать основной пайплайн.

Примеры:

* формат результата;
* возраст ребёнка;
* режим правдивости;
* цель генерации / назначение результата;
* главный персонаж или тема;
* язык пользовательского взаимодействия;
* язык результата;
* количество историй;
* базовый стиль;
* подстиль, если он явно нужен для интерпретации пользовательского запроса.

Если базовый параметр не указан явно, система должна либо:

* извлечь его из контекста с достаточной уверенностью;
* предложить пользователю выбрать значение;
* назначить осознанный дефолт;
* включить значение в один из готовых вариантов интерпретации запроса.

Базовый параметр не должен оставаться пустым к моменту запуска основного пайплайна.

#### 4.3.2 Кастомные жёсткие параметры

Это явно зафиксированные пользовательские требования, которые должны существенно и стабильно влиять на результат.

Примеры:

```text
Главный персонаж — маленький Мук.
```

```text
Он одет в шелка и летает на ковре-самолёте.
```

```text
История должна происходить зимой в лесу.
```

Если такая деталь согласована, система не должна произвольно заменять её на другую.

Например, если в состоянии зафиксировано, что главный персонаж — маленький Мук, то на последующих этапах он не может неожиданно стать золотой рыбкой.

#### 4.3.3 Кастомные мягкие параметры

Это дополнительные пожелания пользователя, которые могут влиять на текст, но не являются обязательными для каждого фрагмента результата.

Примеры:

```text
Можно сделать атмосферу немного волшебной.
```

```text
Хочется, чтобы было уютно.
```

```text
Пусть иногда встречаются смешные слова.
```

Такие параметры сохраняются и могут использоваться при планировании, генерации или редактуре, но не должны ломать базовую структуру запроса.

Кастомные параметры не нужно специально добирать у пользователя. Если пользователь их указал — система фиксирует. Если не указал — генерация идёт без них.

### 4.4 Нормализованный объект параметров

Результатом этапа сбора и нормализации должен быть не только человеко-понятный preview, но и нормализованный объект параметров.

Это не финальная Pydantic-схема и не технический контракт API. На уровне логики документа это промежуточный контракт между:

* пользовательским сценарием;
* prompt-базой;
* агентами уточнения и валидации;
* будущей реализацией оркестратора.

Пример нормализованного объекта:

```json
{
  "content_format": "story",

  "target_age": "3",

Важно не конкретное имя каждого поля, а сама логика: после нормализации у системы должен быть структурированный объект, в котором отделены базовые параметры, жёсткие требования, мягкие предпочтения, уверенность интерпретации и текст preview.

### 4.4.1 User context / история пользователя

На текущем этапе DreamyDraw не использует личный кабинет, профиль ребёнка или историю предыдущих генераций как обязательный источник данных для оркестрации.

Однако архитектура нормализации должна допускать появление дополнительного блока `user_context`. Этот блок может в будущем содержать:

* `user_id` / `session_id`;
* возраст ребёнка по умолчанию;
* предпочитаемый `audience_language`;
* предпочитаемый `result_language`;
* любимые темы или персонажей;
* темы, которые лучше избегать;
* последние сгенерированные серии;
* родительские настройки безопасности;
* историю уточнений и выбранных вариантов.

Минимальный принцип: отсутствие `user_context` не должно мешать генерации. Если контекст есть, он может помогать выбирать дефолты, уточнять неоднозначные запросы и избегать повторов, но не должен молча переопределять явные требования пользователя в текущем запросе.

Явный текущий запрос пользователя имеет приоритет над историей. Например, если в профиле или истории указан возраст 3 года, но пользователь пишет «для ребёнка 5 лет», система должна использовать 5 лет или явно показать, что текущий запрос переопределяет дефолт.

Если в будущем `user_context` будет использовать данные о ребёнке или семейных настройках, он должен подключаться только при наличии разрешённых данных и понятных правил приватности. На уровне текущей логики этот блок является необязательным расширением нормализованного объекта.

---

### 4.4.2 Resolved layers, fallback layers и unresolved details

После анализа запроса и candidate layer resolution система должна различать три типа контекста:

1. **`resolved_layers`** — параметры или сущности, для которых найден конкретный prompt layer / `.md`-файл.
2. **`fallback_layers`** — параметры или сущности, для которых точного слоя нет, но найден ближайший поддерживаемый слой или выбран базовый слой.
3. **`unresolved_details`** — важные детали запроса, для которых нет слоя в prompt-базе, но которые нужно сохранить и передать дальше как freeform context.

Пример:

```text
Пользователь: Расскажи правдивую историю про попугая какаду.
```

Если в базе есть слой про попугая, но нет отдельного слоя про какаду, нормализованный объект может содержать:

```json
{
  "resolved_layers": [
    {
      "type": "character",
      "id": "truth_modes/TRUTH/characters/animals/PARROT",
      "source": "docs/03_PROMPTS/truth_modes/TRUTH/characters/animals/PARROT.md"
    }
  ],
  "fallback_layers": [],
  "unresolved_details": [
    {
      "text": "какаду",
      "type": "animal_subtype",
      "reason": "Нет отдельного prompt layer для какаду",
      "usage": "Передать как freeform factual detail в prompt execution context"
    }
  ]
}
```

Принцип: если prompt layer найден, система использует его детерминированно. Если prompt layer не найден, система не теряет пользовательскую деталь, но и не обещает, что она покрыта базой знаний. Такая деталь передаётся агентам как `unresolved_detail` / `freeform_context`.

Для режима `TRUTH` это особенно важно: фактические детали, которые пришли не из knowledge layer, должны использоваться осторожно и дополнительно проверяться валидатором на отсутствие сомнительных или выдуманных фактов.

### 4.5 Извлечение параметров и оценка уверенности

После получения пользовательского ввода система извлекает возможные параметры генерации и оценивает уверенность по каждому базовому параметру.

Оценка уверенности может использовать условную шкалу от 0 до 100.

Рабочие правила:

| Уверенность | Поведение системы                                                                                     |
| ----------- | ----------------------------------------------------------------------------------------------------- |
| `>= 75`     | Параметр принимается автоматически                                                                    |
| `50–74`     | Система предлагает наиболее вероятную интерпретацию и кратко объясняет, как она повлияет на результат |
| `< 50`      | Система считает параметр неясным и предлагает несколько подходящих вариантов                          |

Пример:

```text
Пользователь: Сделай про ёжика.
```

Возможная оценка:

| Параметр            | Значение           | Уверенность |
| ------------------- | ------------------ | ----------- |
| `content_format`    | `story`            | 80          |
| `main_subject`      | `ёжик`             | 95          |
| `truth_mode`        | неясно             | 30          |
| `utility_mode`      | неясно             | 30          |
| `target_age`        | неясно             | 10          |
| `output_count`      | дефолтное значение | 50          |
| `audience_language` | `ru`               | 95          |
| `result_language`   | `ru`               | 90          |

В таком случае система не должна задавать отдельный вопрос по каждому параметру. Вместо этого она должна сформировать пакетное уточнение с несколькими готовыми вариантами.

---

### 4.6 Пакетное уточнение параметров

Уточняющий диалог должен быть пакетным.

Оркестратор не должен задавать отдельный вопрос по каждому недостающему параметру:

```text
Какой возраст?
Какой режим?
Какой стиль?
Сколько историй?
```

Вместо этого он формирует одно компактное сообщение, в котором:

1. кратко сообщает, как понял запрос;
2. показывает, какие параметры неясны или могут быть выбраны по-разному;
3. предлагает один или несколько полных вариантов интерпретации;
4. даёт пользователю возможность выбрать вариант или уточнить своими словами.

Пакетное уточнение является частью общего уточняющего контура. Этот контур может запускаться не только при неполном или неоднозначном запросе, но и в других случаях:

* пользовательский ввод пустой или бессмысленный;
* запрос содержит противоречие;
* пользователь указал неподдерживаемый стиль как жёсткое требование;
* жёсткие пользовательские детали конфликтуют с выбранным режимом;
* система не может собрать исполнимый набор параметров без дополнительного выбора пользователя.

Пример:

```text
Я понял, что вы хотите истории про ёжика, но можно сделать это по-разному.

Вариант 1 — правдивые короткие истории про ёжика в лесу для ребёнка 3 лет.
Вариант 2 — сказочная серия про маленького ёжика, который ищет друзей.
Вариант 3 — поучительные истории про ёжика, который учится делиться.

Можете выбрать вариант или написать свои уточнения.
```

Каждый предложенный вариант должен быть полным набором базовых параметров. Если пользователь выбирает вариант, система не должна после этого спрашивать базовые параметры, которые сама же могла заполнить в этом варианте.

Уточняющие варианты не являются предварительно сгенерированными историями или готовыми результатами. Это варианты интерпретации пользовательского запроса: каждый вариант описывает, какой запрос система может исполнить и с какими параметрами.

Иными словами, на первом этапе пользователь выбирает не текст, который уже получился, а понятный и исполнимый заказ для генерации.

В будущем интерфейс может показывать рядом с такими вариантами заранее подготовленные best-case examples: короткие примеры текстов, изображения, анимации или карточки режимов. Эти примеры нужны, чтобы объяснить возможности продукта, но они не являются результатом текущей генерации и не должны создаваться на лету в уточняющем контуре.

Best-case examples являются опциональной возможностью пользовательского интерфейса. В CLI они могут быть представлены только как короткие текстовые примеры или ссылки. Полноценный просмотр примеров — изображения, анимации, галереи, карточки режимов — относится к будущему веб-интерфейсу, мобильному приложению, Telegram-боту или другому визуальному клиенту.

Если пользователь отвечает свободным текстом, система снова анализирует ответ, обновляет нормализованный объект параметров и повторно классифицирует состояние запроса. Только после этого система решает, можно ли переходить к candidate layer resolution и финальной валидации.

### 4.7 Лимит уточняющих итераций

Система не должна застревать в бесконечном уточняющем диалоге.

Максимальное количество уточняющих итераций — 5.

Лимит относится ко всему уточняющему контуру, а не только к сценарию неполного или неоднозначного запроса. Он применяется к ситуациям, когда система не может сразу собрать внятный и исполнимый набор параметров:

* неполный запрос;
* неоднозначный запрос;
* пустой или бессмысленный ввод;
* противоречивый запрос;
* неподдерживаемое жёсткое требование;
* конфликт между параметрами или hard details.

После достижения лимита возможны два базовых сценария.

#### Сценарий 1. Есть внятная интерпретация

Если после лимита уточнений у системы есть внятная интерпретация запроса, она может мягко зафиксировать наиболее вероятный набор параметров и передать его на финальную валидацию.

Пример:

```text
Я понял запрос примерно так: подготовить 3 короткие сказочные истории про ёжика для ребёнка 4 лет в спокойном повествовательном стиле.
```

После этого система показывает preview параметров и переходит к основному пайплайну, если финальный валидатор считает набор параметров полным, безопасным, непротиворечивым и исполнимым.

#### Сценарий 2. Нет внятной темы или выбранного варианта

Если после лимита уточнений у системы всё ещё нет внятной темы, выбранного варианта или понятной интерпретации, основной pipeline не запускается.

Это особенно важно для пустого или бессмысленного ввода. Система не должна запускать генерацию полностью на собственных догадках, если пользователь так и не выбрал вариант и не дал понятную тему.

В будущем может быть добавлен отдельный режим `best-effort generation`: система после лимита уточнений сама выбирает безопасные дефолты, предупреждает пользователя и запускает генерацию. Это отдельное продуктовое решение. Его плюс — меньше тупиков для пользователя. Его минус — риск нерелевантного результата и лишних затрат на генерацию.

---

### 4.8 Поведение при пустом или бессмысленном вводе

Если пользователь ничего не ввёл или ввёл бессмысленный текст, система не должна пытаться молча запускать генерацию.

Пустой или бессмысленный ввод рассматривается как вход в общий уточняющий контур, но со специальным первым шагом: система сначала кратко объясняет, что умеет продукт, и предлагает готовые стартовые варианты.

Пример объяснения продукта:

```text
Я могу подготовить серию коротких детских историй: правдивых, сказочных, поучительных или обучающих.
```

Пример стартовых вариантов:

```text
Можно попробовать:

1. 5 сказок про ёжика для ребёнка 4 лет.
2. 3 правдивые истории про белку зимой для ребёнка 3 лет.
3. 5 поучительных историй про маленького Мука, который учится быть смелым.
```

В отличие от обычного понятного запроса, при пустом или бессмысленном вводе система не должна начинать генерацию без выбора или подтверждения пользователя. Если система сама предложила исходный вариант, пользователь должен выбрать его или уточнить.

После выбора варианта система не переходит сразу к финальной валидации. Сначала выбранный вариант проходит повторный анализ и нормализацию, затем повторную классификацию состояния параметров.

Для пустого или бессмысленного ввода правило осмысленного выбора имеет приоритет над общим лимитом уточнений: система может предложить варианты, но не должна запускать генерацию полностью на собственных догадках, если пользователь так и не выбрал вариант и не дал внятную тему.

Отдельный близкий случай — мета-запрос о самом продукте или агенте, например:

```text
Расскажи о себе.
```

Такой ввод не считается запросом на генерацию историй. Система должна распознать его как просьбу объяснить, что умеет DreamyDraw, кратко рассказать о возможностях продукта и при необходимости предложить стартовые варианты. Основной pipeline в этом случае не запускается, пока пользователь не выберет вариант или не сформулирует внятную тему генерации.

---

### 4.9 Проверка согласованности параметров

Перед запуском основного пайплайна система должна проверить, что параметры не противоречат друг другу.

Противоречие может возникнуть между:

* режимом правдивости и сюжетной деталью;
* возрастом ребёнка и сложностью темы;
* utility-режимом и выбранным стилем;
* жёстким пользовательским требованием и правилами безопасности;
* выбранным подстилем и режимом генерации.

Пример противоречия:

```text
Расскажи правдивую поучительную историю про ёжика на ковре-самолёте.
```

Здесь конфликтуют:

```text
truth_mode = Правда
сюжетная деталь = ёжик летает на ковре-самолёте
```

Для режима `Правда` ковёр-самолёт как реальное событие не подходит. Система не должна молча игнорировать деталь или незаметно менять режим. Она должна явно сообщить о противоречии и предложить непротиворечивые варианты.

Пример ответа:

```text
В запросе есть противоречие: режим “Правда” предполагает реалистичную историю, а ковёр-самолёт — сказочный элемент.

Можно выбрать один из вариантов:

1. Оставить режим “Правда”: поучительная реалистичная история про ёжика в лесу.
2. Переключить в режим “Сказка”: история про ёжика, который летает на ковре-самолёте.
3. Сохранить правдивый режим, но перенести ковёр-самолёт в воображение ребёнка или рисунок.
```

Каждая альтернатива должна:

* быть внутренне непротиворечивой;
* сохранять максимум исходного намерения пользователя;
* содержать полный набор базовых параметров;
* быть исполнимой через доступные режимы и prompt layers.

---

### 4.10 Prompt-aware интерпретация запроса

Оркестратор не должен интерпретировать пользовательский запрос в отрыве от доступной базы режимов, стилей, подстилей и prompt layers.

Если система предлагает пользователю готовую интерпретацию запроса, эта интерпретация должна быть исполнимой: для неё должны существовать соответствующие режимы, описания, fallback-слои или prompt layers.

Prompt layers участвуют в логике первого блока в двух разных режимах.

#### 4.10.1 Prompt-aware metadata lookup

На этапе анализа и уточнения запроса система использует metadata prompt layers, чтобы:

* понять, какие режимы, стили и подстили доступны;
* сопоставить пользовательские формулировки с поддерживаемыми слоями;
* не предлагать пользователю неподдерживаемую стилизацию как гарантированную;
* определить возможные fallback-варианты;
* подготовить человеко-понятное объяснение для уточнения или preview.

На этом этапе агент не обязан использовать полный текст prompt layer. Достаточно metadata: названия, описания, ограничений, `good_for`, `bad_for`, примеров и других кратких сведений, которые помогают понять, что система реально поддерживает.

Пример:

```text
Хочу сказку как у Чуковского про ёжика.
```

Система должна сопоставить запрос с доступными параметрами:

```text
truth_mode = Сказка
utility_mode = Повествовательный
substyle = CHUKOVSKY
main_subject = ёжик
```

Для общения с пользователем достаточно краткого описания режима или стиля:

```text
Я понял так: это будет сказочная игровая история про ёжика, с ритмичной речью, повторами и лёгкой абсурдностью в духе детской поэзии.
```

#### 4.10.2 Candidate layer resolution

До preview система должна выполнить candidate layer resolution: определить, какие prompt layers или fallback-слои соответствуют нормализованным параметрам.

Цель этого шага — убедиться, что preview не обещает пользователю неподдерживаемую стилизацию, режим или формат результата.

Если точный слой найден, он фиксируется как candidate layer.
Если точного слоя нет, но есть близкий поддерживаемый слой, система фиксирует fallback-кандидат или предлагает его пользователю, если исходное требование было жёстким.
Если точного и близкого слоя нет, система использует базовый слой и сохраняет пользовательскую деталь в `hard_details` или `soft_preferences`, в зависимости от силы требования.

Главный принцип:

```text
Система предлагает пользователю только те варианты, которые может исполнить через текущую prompt-базу.
```

Preview должен показывать уже исполнимую интерпретацию, а не предварительную догадку, которая может измениться после загрузки `.md`-файлов.

После preview происходит уже не принятие решения о том, что обещать пользователю, а подготовка prompt execution context: фиксация выбранных layer ids, fallback-решений, общего контекста параметров, hard constraints и soft preferences для основного pipeline.

Фактическая загрузка и сборка `.md` prompt bodies выполняется уже внутри основного pipeline по мере необходимости конкретного агента или этапа.

### 4.11 Fallback-логика prompt layers

Если пользователь указывает стиль, подстиль, персонажа или другую деталь, для которой нет точного prompt layer, система не должна падать и не должна молча терять эту деталь.

Возможны три базовых сценария:

1. **Есть точный prompt layer**
   Система использует соответствующий слой.

2. **Точного prompt layer нет, но есть близкий поддерживаемый слой**
   Система может предложить пользователю ближайший поддерживаемый вариант.

3. **Нет точного или близкого слоя**
   Система использует базовый prompt layer, но сохраняет пользовательскую деталь как контекст — в `hard_details` или `soft_preferences`, в зависимости от силы требования.

Дополнительное правило: неподдерживаемый стиль или подстиль не всегда требует уточнения. Если пользователь сформулировал его как жёсткое требование, система должна уточнить или предложить поддерживаемую альтернативу. Если это мягкое пожелание, система может не останавливать сценарий, а сохранить пожелание в `soft_preferences` и продолжить на базовом или ближайшем поддерживаемом слое.

Пример:

```text
Пользователь: Сделай историю в стиле X про ёжика.
```

Если слоя `X` нет, система не должна отвечать так, будто стиль `X` полноценно поддерживается. Она должна либо предложить ближайшую поддерживаемую альтернативу, либо использовать базовый стиль и честно сохранить пожелание как дополнительную деталь.

Логика выбора:

```text
точный слой найден
        → использовать точный prompt layer

точный слой не найден, есть близкий слой
        → предложить близкий поддерживаемый вариант

точный слой не найден, близкого слоя нет
        → если требование жёсткое: уточнить или предложить альтернативу
        → если пожелание мягкое: использовать базовый prompt layer
        → сохранить исходную деталь в состоянии
        → не обещать пользователю точную стилизацию, которую система не поддерживает
```

Fallback-логика должна сохранять максимум исходного намерения пользователя, но не должна создавать ложное впечатление, что система поддерживает недоступный режим или стиль.

### 4.12 Самоописываемые prompt layers

Параметры генерации, режимы, стили и подстили будут развиваться. В системе будут появляться новые значения, новые режимы, новые возрастные профили, новые utility-режимы и новые prompt layers.

Поэтому каждый prompt layer, который система может использовать или предложить пользователю, должен быть самоописываемым.

Рекомендуемый формат — один `.md` файл, содержащий:

1. YAML metadata-блок;
2. основной prompt body.

Пример:

```markdown
---
id: CHUKOVSKY
name: Чуковский
type: substyle
truth_mode: FAIRY_TALE
user_description: Игровой, ритмичный, немного абсурдный стиль для детских сказочных историй.
short_description: Ритм, повторы, игра слов, динамика, лёгкая сказочная нелепость.
constraints:
  - Не копировать конкретные тексты.
  - Не делать лексику слишком сложной для возраста 3–5 лет.
  - Не перегружать текст абсурдом.
good_for:
  - сказочные истории
  - игровые истории
  - короткие ритмичные тексты
bad_for:
  - строгий режим правды
  - натуралистичные истории
  - энциклопедический стиль
examples:
  - Сказочная история про ёжика с повторами и весёлыми словами.
  - Игровая история с быстрым ритмом и простыми образами.
example_result_ids:
  - fairy_tale_chukovsky_hedgehog_best_case_01
sample_text: Короткий заранее подготовленный пример, показывающий общий характер стиля.
sample_image_url: null
sample_animation_url: null
---

# Основной prompt layer

...
```

YAML metadata используется на этапе понимания и уточнения запроса. Основной markdown prompt body используется на этапе генерации, валидации или редактуры.

Поля `example_result_ids`, `sample_text`, `sample_image_url` и `sample_animation_url` являются опциональными. Они могут использоваться визуальными интерфейсами для показа заранее подготовленных best-case examples. Оркестрация не должна зависеть от наличия этих полей: если примеров нет, система всё равно должна уметь анализировать запрос, уточнять параметры и запускать основной pipeline.

Такой подход позволяет:

* держать описание и prompt layer в одном месте;
* не допускать расхождения между тем, что система обещает пользователю, и тем, что реально делает генератор;
* валидировать prompt-базу автоматически;
* расширять систему новыми режимами и стилями без переписывания логики оркестратора;
* давать пользователю человеко-понятные объяснения выбранных параметров.

### 4.13 Финальная валидация параметров

Перед preview и запуском основного пайплайна финальный валидатор должен проверить, что набор параметров:

* полный;
* внятный;
* непротиворечивый;
* безопасный для детского контента;
* исполнимый с учётом доступных режимов и prompt layers;
* не содержит конфликта между hard details и выбранными режимами;
* соответствует возрасту ребёнка;
* не обещает неподдерживаемый формат, стиль или подстиль.

Финальная валидация отличается от анализа и нормализации.

Анализ и повторный анализ отвечают на вопрос:

```text
Что пользователь имел в виду и какие параметры можно извлечь из его ввода?
```

Финальная валидация отвечает на другой вопрос:

```text
Можно ли с этим нормализованным объектом запускать основной pipeline?
```

За финальным валидатором остаётся последнее решение о переходе к preview и основному pipeline.

Даже если пользователь выбрал один из предложенных вариантов или уточнил запрос свободным текстом, система всё равно должна провести повторный анализ, повторную классификацию состояния параметров и финальную проверку.

Упрощённая схема:

```text
Выбор / уточнение пользователя
        │
        ▼
Повторный анализ и нормализация
        │
        ▼
Повторная классификация состояния параметров
        │
        ├─ полный и понятный запрос
        │       │
        │       ▼
        │   Candidate layer resolution
        │       │
        │       ▼
        │   Финальная валидация параметров
        │
        └─ всё ещё есть проблема
                │
                ▼
            уточняющий контур / альтернативы / STOP
```

Если финальная валидация проходит успешно, система переходит к preview.

Если финальная валидация не проходит, система не должна автоматически запускать основной pipeline. Сначала нужно повторно классифицировать причину отказа:

* неполнота параметров;
* противоречие;
* неподдерживаемое жёсткое требование;
* конфликт с возрастом;
* конфликт с безопасностью;
* невозможность собрать исполнимый набор prompt layers.

После этого система либо возвращается в уточняющий контур, либо предлагает альтернативы, либо останавливает сценарий, если запрос нельзя привести к исполнимому виду.

---

### 4.14 Preview параметров перед запуском генерации

После успешной финальной валидации система показывает пользователю friendly-preview параметров будущей генерации.

Preview не требует обязательного действия от пользователя. Его цель — дать пользователю коротко увидеть, как система поняла запрос, перед запуском основного пайплайна.

Пример:

```text
Я подготовлю серию историй:

Формат: истории
Режим: Правда
Тип: Повествовательные истории
Возраст: 3 года
Главный персонаж: ёжик
Место: зимний лес
Базовый стиль: спокойный
Подстиль: натуралистичный
Количество: 5 историй
Язык общения: русский
Язык результата: русский
```

Preview может отображаться короткое время, например 3–5 секунд, но это пример UX-реализации, а не логическое требование. В CLI, веб-интерфейсе, Telegram-боте или другом интерфейсе этот шаг может выглядеть по-разному.

В будущем может быть добавлена возможность отменить запуск или изменить параметры в течение preview-окна. На текущем уровне логики это считается дополнительной UX-возможностью, а не обязательным условием запуска.

Preview параметров не является preview готового результата. Он не показывает уже сгенерированные истории, изображения или анимации. Его задача — подтвердить исполнимую интерпретацию запроса перед запуском основного pipeline.

Визуальный интерфейс может дополнительно показывать заранее подготовленные best-case examples рядом с preview или вариантами уточнения, если такие примеры связаны с выбранными режимами и prompt layers через metadata. Такие примеры должны восприниматься как демонстрация возможностей продукта, а не как обещание конкретного результата текущей генерации.

Preview должен показываться всегда перед основным пайплайном, независимо от того, был ли исходный запрос полным, неполным, противоречивым или выбранным из предложенных вариантов.

### 4.16 Переход к основному пайплайну

После preview система переходит к основному пайплайну генерации.

К этому моменту должны быть выполнены условия:

1. базовые параметры заполнены;
2. жёсткие пользовательские требования зафиксированы;
3. противоречия устранены;
4. финальный валидатор разрешил переход;
5. пользователь увидел preview параметров.

Дальнейшая логика основного пайплайна включает:

* подготовку `prompt execution context`;
* генерацию пула текстовых кандидатов;
* проверку разнообразия тем;
* hard gates и scoring;
* ранжирование кандидатов;
* последовательную валидацию и редактуру;
* набор итогового комплекта `approved texts`;
* fallback при нехватке approved texts;
* выход второго этапа для дальнейшей генерации иллюстраций.

Эти этапы описываются в разделе 5.

## 5. Основной pipeline подготовки результата

Второй этап отвечает за подготовку итогового набора текстов, которые затем могут быть переданы в следующий этап — генерацию иллюстраций.

На вход второго этапа приходит уже согласованная и провалидированная интерпретация пользовательского запроса. На этом этапе система не должна заново решать, что обещать пользователю. Она должна исполнить выбранную интерпретацию: сгенерировать пул текстовых кандидатов, оценить их, отобрать лучшие, провалидировать и отредактировать до финального набора.

---

### 5.1 Короткая обзорная схема

```text
Нормализованные параметры
        │
        ▼
Prompt execution context
        │
        ▼
Генерация candidate texts
        │
        ▼
Scoring / ranking
        │
        ▼
Последовательная валидация и редактура
        │
        ▼
Approved texts
        │
        ▼
Переход к генерации изображений
```

Ключевая идея второго этапа: система готовит не один финальный набор сразу, а запас текстовых кандидатов. Затем кандидаты проходят оценку, ранжирование, последовательную валидацию и редактуру, пока не будет набрано нужное количество `approved texts`.

---

### 5.2 Детальная схема отбора текстов

```text
[1] Prompt execution context
        │
        ▼
[2] Генерация пула candidate texts
        │
        ├─ topic
        ├─ text
        ├─ intended utility
        └─ expected visual idea
        │
        ▼
[3] Проверка разнообразия тем
        │
        ├─ тема новая → дальше
        └─ тема похожа на предыдущие → заменить / понизить / отбросить
        │
        ▼
[4] Hard gates
        │
        ├─ safety
        ├─ соответствие возрасту
        ├─ соответствие truth_mode
        ├─ сохранение hard_details
        └─ соответствие utility goal
        │
        ▼
[5] Scoring
        │
        ├─ интересность для ребенка
        ├─ возрастная уместность
        ├─ стиль
        ├─ поучительность / narrative fit
        ├─ визуальный потенциал
        └─ разнообразие
        │
        ▼
[6] Ранжирование candidates
        │
        ▼
[7] Последовательная валидация по рейтингу
        │
        ├─ passed → approved texts
        ├─ needs revision → refiner, максимум 2 попытки
        └─ failed → следующий candidate
        │
        ▼
[8] Набран output_count?
        │
        ├─ да → выход второго этапа
        └─ нет → fallback: regenerate / предложить безопасные варианты / STOP
```

Эта схема описывает бизнес-логику отбора. Она не фиксирует, сколько именно LLM-вызовов или отдельных агентов будет использовано в реализации.

---

### 5.3 Входные данные второго этапа

На вход второго этапа приходит результат первого этапа:

* нормализованные параметры (`content_format`, `truth_mode`, `utility_mode`, `target_age`, `output_count`, языки, стиль, subject/details);
* `prompt_context`:
  * `resolved_layers`;
  * `fallback_layers`;
  * `unresolved_details`;
* `user_context`, если он доступен;
* `hard_details`;
* `soft_preferences`;
* ограничения безопасности и возрастной уместности.

Второй этап должен использовать эти данные как рамку исполнения. Если в `prompt_context` есть unresolved details, они не превращаются в несуществующие prompt layers, но передаются генератору как дополнительный контекст.

Пример:

```text
Запрос: Расскажи правдивые истории про попугая какаду.

resolved_layers:
  - TRUTH/characters/animals/birds/PARROT.md

unresolved_details:
  - factual_detail: "какаду"
```

В этом случае система использует слой про попугая, если он есть, а деталь про какаду передает как freeform factual detail.

---

### 5.4 Prompt execution context

`Prompt execution context` — это рабочий набор данных, с которым основной pipeline начинает исполнение. Он не означает, что система заранее собирает все prompt bodies для всех будущих агентов.

На этом шаге фиксируются:

* выбранные layer ids;
* fallback-решения;
* unresolved details;
* hard constraints;
* soft preferences;
* возрастные ограничения;
* utility goal;
* языковые параметры;
* общие требования к структуре результата.

Фактическая загрузка и сборка `.md` prompt bodies может происходить поэтапно внутри конкретных частей pipeline: генерации кандидатов, scoring, валидации, редактуры и подготовки к иллюстрациям.

Главный принцип: до генерации candidate texts система уже знает, какие слои и ограничения должны участвовать в исполнении, но не обязана заранее материализовать все промпты для всех будущих шагов.

---

### 5.5 Генерация пула candidate texts

Второй этап генерирует запас кандидатов больше, чем итоговое количество, запрошенное пользователем.

Для MVP:

```text
candidate_count = 20
```

В будущем `candidate_count` может стать функцией от `output_count`.

Пример будущей логики:

```text
output_count = 3  → candidate_count = 8
output_count = 5  → candidate_count = 10
output_count = 10 → candidate_count = 15-20
```

Каждый кандидат должен иметь минимум:

```json
{
  "candidate_id": "c01",
  "topic": "Ёжик ищет следы на снегу",
  "text": "Короткий текст истории...",
  "questions": ["..."],
  "intended_utility": "NARRATIVE",
  "expected_visual_idea": "ёжик у следов на снегу",
  "used_context": {
    "resolved_layers": [],
    "fallback_layers": [],
    "unresolved_details": []
  }
}
```

Тема обязательна. Она нужна для проверки разнообразия и для запрета редактору менять смысл кандидата.

---

### 5.6 Разнообразие и дедупликация тем

Кандидаты не должны повторять одну и ту же тему.

Во время генерации или пост-проверки кандидат `N` должен учитывать темы предыдущих кандидатов:

```text
candidate 1 → тема 1
candidate 2 → знает тему 1 и не повторяет ее
candidate 3 → знает темы 1-2 и не повторяет их
...
candidate 20 → знает темы 1-19 и не повторяет их
```

На уровне бизнес-логики важно не то, будет ли это реализовано одним LLM-вызовом, несколькими вызовами или отдельной проверкой. Важно, что итоговый пул кандидатов должен стремиться к разнообразию тем.

Если тема похожа на предыдущие, система может:

* заменить кандидата;
* понизить его novelty score;
* отбросить его до этапа ранжирования.

---

### 5.7 Scoring и hard gates

После генерации кандидаты проходят оценку.

Оценка состоит из двух уровней.

#### 5.7.1 Hard gates

Hard gate — это проверка, при провале которой кандидат нельзя считать обычным хорошим кандидатом, даже если он интересный.

Примеры hard gates:

* safety;
* грубый конфликт с `truth_mode`;
* грубое несоответствие возрасту;
* нарушение обязательных `hard_details`;
* невозможность сохранить `main_subject`;
* невозможность сохранить utility goal.

Если кандидат провалил hard gate, он попадает в rejected pool и не участвует в обычном ранжировании.

#### 5.7.2 Score components

Если hard gates пройдены, кандидат получает частные оценки:

* `child_interest`;
* `age_fit`;
* `utility_fit`;
* `style_fit`;
* `truth_fit`;
* `novelty`;
* `visual_potential`.

После этого считается общий score, например как взвешенное среднее. Если значимый параметр получил очень низкую оценку, кандидат может быть отклонен без суммирования общего score.

Для MVP оценку может ставить один агент. В будущем разные score components могут оцениваться разными агентами или детерминированными проверками.

---

### 5.8 Ранжирование кандидатов

Кандидаты сортируются по итоговому score.

Цель ранжирования — сначала валидировать и редактировать наиболее перспективные тексты, а не весь пул.

```text
ranked_candidates = candidates
  .filter(hard_gates_passed)
  .sort(by total_score desc)
```

Кандидаты, полностью провалившие значимые hard gates, не должны попадать в обычный список лучших вариантов.

---

### 5.9 Последовательная валидация и редактура

Система идет по ranked candidates сверху вниз и набирает итоговый список approved texts.

```text
approved = []

for candidate in ranked_candidates:
    validation_result = validate(candidate)

    if validation_result == passed:
        approved.append(candidate)

    if validation_result == needs_revision:
        refined = refine_same_topic(candidate)
        validate_again(refined)

    if validation_result == failed:
        skip_candidate

    stop when len(approved) == output_count
```

Валидация и редактура одного кандидата ограничены.

Для MVP:

```text
max_refine_attempts_per_candidate = 2
```

Редактор может:

* исправлять факты;
* упрощать формулировки;
* адаптировать текст под возраст;
* улучшать вопросы;
* усиливать соответствие utility goal;
* убирать нарушения safety / truth constraints.

Редактор не должен:

* менять тему кандидата;
* менять `main_subject`;
* менять utility goal;
* удалять обязательные `hard_details`;
* превращать кандидата в другой сюжет.

Если без смены темы исправить кандидата нельзя, кандидат пропускается.

---

### 5.10 Набор итогового комплекта approved texts

Основной успешный сценарий:

```text
approved_count == output_count
```

Каждый approved text должен содержать:

* тему;
* финальный текст;
* вопросы;
* служебную информацию о пройденной валидации;
* ссылки на использованный prompt context / trace metadata, если это нужно для отладки.

Компактная таблица состояний:

| Шаг | Что фиксируется |
| --- | --- |
| Candidate generation | `candidate_id`, `topic`, `text`, `questions`, `expected_visual_idea` |
| Diversity check | похожесть темы, novelty signal |
| Hard gates | passed / failed по критичным ограничениям |
| Scoring | score components и общий score |
| Validation | passed / needs_revision / failed |
| Refinement | исправленный текст без смены темы |
| Approval | финальный approved text |

---

### 5.11 Fallback при нехватке approved texts

Редкий, но возможный сценарий:

```text
approved_count < output_count
```

В этом случае система не должна показывать пользователю кандидатов, которые полностью провалили значимые hard gates, особенно safety.

Базовая MVP-логика:

1. Если есть safe fallback candidates, система может предложить пользователю выбрать из 2-3 лучших вариантов без свободного редактирования.
2. Если safe fallback candidates нет, система предлагает перегенерировать candidate pool.
3. Если повторная генерация тоже не дала достаточно кандидатов, сценарий останавливается с понятным сообщением.

Эту ситуацию нужно логировать для анализа.

Пример служебной записи:

```json
{
  "candidate_pool_failure": {
    "reason": "not_enough_valid_candidates",
    "requested": 5,
    "approved": 2,
    "hard_gate_failures": {
      "safety": 1,
      "truth_fit": 6,
      "age_fit": 3
    }
  }
}
```

В будущем можно добавить более умный пользовательский арбитраж или возможность дать комментарий, но для MVP лучше ограничиться выбором готовых safe fallback candidates или перегенерацией.

---

### 5.12 Выход второго этапа

Результат второго этапа — набор `approved texts`, готовый к передаче в следующий этап.

На выходе должно быть:

* `approved_texts`;
* темы approved texts;
* вопросы для разговора с ребенком;
* `prompt_context` / trace metadata;
* ограничения и детали, важные для будущей генерации иллюстраций;
* информация о fallback-решениях, если они применялись.

Следующий этап — генерация иллюстраций — должен использовать утвержденные тексты и темы как входные данные. Он не должен менять финальный текст, тему или смысл approved result.
