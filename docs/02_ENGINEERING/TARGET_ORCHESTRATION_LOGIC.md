# TARGET_ORCHESTRATION_LOGIC.md

# Целевая логика оркестрации

## 1. Назначение документа

Документ описывает целевую логику работы оркестратора с точки зрения пользовательского сценария, подготовки параметров генерации, принятия решений системой, взаимодействия с пользователем и последовательной подготовки итогового результата.

Документ фиксирует не техническую реализацию LangGraph-графа, а логическую модель продукта:

* что пользователь вводит;
* как система понимает и нормализует запрос;
* какие параметры должна собрать перед запуском генерации;
* когда система может продолжать автоматически;
* когда должна уточнить запрос;
* как обрабатываются неполные, пустые, бессмысленные или противоречивые запросы;
* как формируется и валидируется набор параметров для дальнейшей генерации;
* как пользователь видит итоговую интерпретацию своего запроса перед запуском основного пайплайна;
* как после фиксации параметров система подбирает prompt layers / `.md`-файлы;
* как дальше организуется основной процесс подготовки результата: планирование, валидация, редактура, генерация текстов, проверка текстов, генерация иллюстраций и финальная выдача.

На текущем этапе документа подробно описан первый крупный блок — сбор и нормализация параметров генерации. Остальные этапы основного пайплайна пока обозначаются верхнеуровнево и будут детализироваться в следующих итерациях документа.

Техническая реализация — конкретные ноды LangGraph, routing-функции, структура `SessionState`, interrupt/resume и Langfuse-наблюдаемость — должна описываться отдельно в технической спецификации. Этот документ является логическим основанием для такой реализации.

---

## 2. Общий принцип целевой оркестрации

На текущем уровне детализации в целевой оркестрации предварительно выделяются два крупных блока:

1. **Сбор, нормализация и валидация параметров генерации**
   Система получает произвольный пользовательский ввод и превращает его в внятный, полный, непротиворечивый и исполнимый набор параметров.

2. **Основной пайплайн подготовки результата**
   После фиксации параметров система подбирает соответствующие prompt layers / `.md`-файлы, формирует план серии, валидирует и улучшает его, генерирует тексты, проверяет и редактирует их, готовит иллюстрации и отдаёт пользователю финальный результат.

Это деление не является окончательным. По мере детализации документа количество крупных этапов может измениться: некоторые блоки могут быть выделены отдельно, объединены или разложены на несколько самостоятельных логических контуров.

Ключевой принцип: основной пайплайн генерации не должен запускаться, пока система не получила минимально необходимый набор базовых параметров и не признала его достаточно понятным, согласованным и исполнимым.

## 3. Верхнеуровневая схема целевой оркестрации

```text
[0] Пользовательский ввод
        │
        ▼
[1] Первичный анализ запроса
        │
        ├─ извлечение возможных параметров
        ├─ оценка полноты базовых параметров
        ├─ оценка уверенности по базовым параметрам
        ├─ выделение кастомных жёстких и мягких деталей
        └─ первичная проверка исполнимости через доступные режимы / prompt layers
        │
        ▼
[2] Классификация ситуации
        │
        ├─ запрос полный и понятный
        │       │
        │       ▼
        │   [5] Финальная валидация параметров
        │
        ├─ запрос неполный или неоднозначный
        │       │
        │       ▼
        │   [3] Пакетное уточнение параметров
        │       │
        │       ├─ пользователь выбрал готовый вариант
        │       ├─ пользователь уточнил свободным текстом
        │       └─ достигнут лимит уточнений
        │
        ├─ запрос пустой или бессмысленный
        │       │
        │       ▼
        │   [4] Объяснение продукта + готовые стартовые варианты
        │       │
        │       ├─ пользователь выбрал вариант
        │       └─ пользователь уточнил свободным текстом
        │
        └─ запрос противоречивый
                │
                ▼
            [3a] Объяснение противоречия + непротиворечивые альтернативы
                │
                ├─ пользователь выбрал альтернативу
                └─ пользователь уточнил свободным текстом

[3] / [3a] / [4]
        │
        ▼
[1] Повторный анализ обновлённого запроса или выбранного варианта
        │
        ▼
[5] Финальная валидация параметров
        │
        ├─ pass
        │     │
        │     ▼
        │ [6] Preview параметров перед запуском
        │     │
        │     ▼
        │ [7] Подбор prompt layers / `.md`-файлов
        │     │
        │     ▼
        │ [8] Основной пайплайн подготовки результата
        │
        └─ fail
              │
              ├─ возврат к пакетному уточнению
              ├─ предложение альтернатив
              └─ остановка, если запрос нельзя привести к исполнимому виду
```

Основной пайплайн подготовки результата на текущем этапе документа обозначается верхнеуровнево:

```text
[8] Основной пайплайн подготовки результата
        │
        ├─ планирование серии
        ├─ валидация плана
        ├─ редактура / уточнение плана
        ├─ генерация текстов
        ├─ валидация и редактура текстов
        ├─ возможное согласование с пользователем
        ├─ генерация иллюстраций
        └─ финальная выдача результата
```

Подробная логика этих этапов будет описываться дальше по мере развития документа.

## 4. Сбор и нормализация параметров генерации

### 4.1 Цель этапа

Цель этапа — превратить произвольный пользовательский ввод в минимально достаточный, нормализованный и согласованный набор параметров, на основе которого можно запускать основной пайплайн генерации.

Пользователь не обязан заполнять форму или заранее знать все режимы системы. Он может написать запрос свободным текстом, неполно, нестрого или даже оставить поле пустым. Задача оркестратора — помочь пользователю прийти к исполнимому запросу без лишнего трения.

Пример полного запроса:

```text
Сделай 5 коротких натуралистичных историй про ёжика зимой в лесу для ребёнка 3 лет, в спокойном повествовательном стиле.
```

Пример неполного запроса:

```text
Сделай про ёжика.
```

Пример противоречивого запроса:

```text
Расскажи правдивую поучительную историю про ёжика на ковре-самолёте.
```

Пример пустого или бессмысленного ввода:

```text
...
```

или

```text
ываыва 123
```

Во всех случаях система должна прийти к одному из двух результатов:

1. сформирован полный и согласованный набор параметров;
2. пользователь отказался продолжать или не выбрал ни один из предложенных вариантов.

---

### 4.2 Базовые обязательные параметры

Перед запуском основного пайплайна должны быть заполнены базовые параметры генерации.

На старте целевой логики базовыми считаются:

| Параметр            | Назначение                                        | Пример                                                            |
| ------------------- | ------------------------------------------------- | ----------------------------------------------------------------- |
| `content_format`    | Формат результата                                 | `story`                                                           |
| `truth_mode`        | Режим отношения к реальности                      | `Правда`, `Миф`, `Сказка`                                         |
| `utility_mode`      | Цель генерации / назначение результата            | `Повествовательный`, `Поучительный`, `Английский`                 |
| `target_age`        | Возраст ребёнка                                   | `3`, `3.5`, `4`, `4.5`, `5`                                       |
| `main_subject`      | Главный персонаж, объект или тема                 | `ёжик`, `маленький Мук`, `белка зимой`                            |
| `output_count`      | Количество историй / элементов результата         | `3`, `5`, `7`                                                     |
| `audience_language` | Язык пользовательского взаимодействия             | `ru`                                                              |
| `result_language`   | Язык итогового результата                         | `ru`, в будущем — `en` и другие                                   |
| `text_style_base`   | Базовый тон / стиль текста                        | `спокойный`, `познавательный`, `игровой`                          |
| `substyle`          | Более конкретный подстиль или жанровая стилизация | `натуралистичный`, `русско-народный`, `в духе Чуковского`, `null` |

Часть параметров имеет ограниченный набор допустимых значений. Например, `content_format`, `truth_mode`, `utility_mode`, `target_age`, `audience_language` и `result_language` должны быть выбраны из заранее известных значений системы.

Другие параметры могут быть более свободными, но всё равно должны быть нормализованы. Например, пользователь может написать «как у Чуковского», а система должна интерпретировать это как конкретный поддерживаемый подстиль, если такой prompt layer существует.

`substyle` является базовым полем нормализованного объекта, но не всегда обязательным значением. Оно может быть `null`, если пользователь не указал конкретный подстиль, если подстиль не нужен для выбранного формата результата или если точный подстиль не найден в prompt-базе. В таком случае система использует базовые слои стиля и сохраняет пользовательские пожелания в `hard_details` или `soft_preferences`, если они важны для дальнейшей генерации.

На старте продукт ориентирован на русскоязычную аудиторию:

* `audience_language = ru`;
* уточнения, preview и объяснения показываются по-русски;
* `utility_mode = ENGLISH` не означает автоматически, что весь интерфейс или весь результат должен быть на английском;
* `utility_mode = ENGLISH` означает образовательную цель, связанную с изучением английского;
* язык итогового результата должен развиваться отдельно через `result_language` или аналогичное поле.

Поле `content_format` фиксирует формат результата. На старте основным форматом считается:

```text
content_format = story
```

В будущем рядом могут появиться другие форматы, например:

* `picture_riddle`;
* `english_card`;
* `storyboard`;
* `dialogue_practice`.

Добавление новых форматов не должно требовать пересборки всей логики сбора параметров. Формат результата должен становиться одним из базовых параметров, который влияет на дальнейший подбор prompt layers и основной pipeline.

### 4.3 Типы параметров

Параметры генерации делятся на три группы.

#### 4.3.1 Базовые обязательные параметры

Это параметры, без которых нельзя запускать основной пайплайн.

Примеры:

* формат результата;
* возраст ребёнка;
* режим правдивости;
* цель генерации / назначение результата;
* главный персонаж или тема;
* язык пользовательского взаимодействия;
* язык результата;
* количество историй;
* базовый стиль;
* подстиль, если он явно нужен для интерпретации пользовательского запроса.

Если базовый параметр не указан явно, система должна либо:

* извлечь его из контекста с достаточной уверенностью;
* предложить пользователю выбрать значение;
* назначить осознанный дефолт;
* включить значение в один из готовых вариантов интерпретации запроса.

Базовый параметр не должен оставаться пустым к моменту запуска основного пайплайна.

#### 4.3.2 Кастомные жёсткие параметры

Это явно зафиксированные пользовательские требования, которые должны существенно и стабильно влиять на результат.

Примеры:

```text
Главный персонаж — маленький Мук.
```

```text
Он одет в шелка и летает на ковре-самолёте.
```

```text
История должна происходить зимой в лесу.
```

Если такая деталь согласована, система не должна произвольно заменять её на другую.

Например, если в состоянии зафиксировано, что главный персонаж — маленький Мук, то на последующих этапах он не может неожиданно стать золотой рыбкой.

#### 4.3.3 Кастомные мягкие параметры

Это дополнительные пожелания пользователя, которые могут влиять на текст, но не являются обязательными для каждого фрагмента результата.

Примеры:

```text
Можно сделать атмосферу немного волшебной.
```

```text
Хочется, чтобы было уютно.
```

```text
Пусть иногда встречаются смешные слова.
```

Такие параметры сохраняются и могут использоваться при планировании, генерации или редактуре, но не должны ломать базовую структуру запроса.

Кастомные параметры не нужно специально добирать у пользователя. Если пользователь их указал — система фиксирует. Если не указал — генерация идёт без них.

### 4.4 Нормализованный объект параметров

Результатом этапа сбора и нормализации должен быть не только человеко-понятный preview, но и нормализованный объект параметров.

Это не финальная Pydantic-схема и не технический контракт API. На уровне логики документа это промежуточный контракт между:

* пользовательским сценарием;
* prompt-базой;
* агентами уточнения и валидации;
* будущей реализацией оркестратора.

Пример нормализованного объекта:

```json
{
  "content_format": "story",
  "truth_mode": "TRUTH",
  "utility_mode": "NARRATIVE",
  "target_age": "3",
  "output_count": 5,
  "audience_language": "ru",
  "result_language": "ru",
  "main_subject": "ёжик",
  "subjects": ["ёжик"],
  "setting": {
    "place": "лес",
    "season": "зима",
    "time": null
  },
  "text_style_base": "calm",
  "substyle": "naturalistic",
  "hard_details": [
    "главный персонаж — ёжик",
    "действие происходит зимой в лесу",
    "истории должны быть реалистичными"
  ],
  "soft_preferences": [
    "мягкий тон",
    "простые фразы",
    "без страшных сцен"
  ],
  "confidence": {
    "content_format": 90,
    "truth_mode": 85,
    "utility_mode": 80,
    "target_age": 95,
    "main_subject": 95,
    "output_count": 90,
    "audience_language": 95,
    "result_language": 95,
    "text_style_base": 75,
    "substyle": 80
  },
  "requires_clarification": false,
  "clarification_reason": null,
  "preview_text": "Я подготовлю 5 спокойных правдивых историй про ёжика зимой в лесу для ребёнка 3 лет. Тексты будут реалистичными, простыми и подходящими для чтения ребёнку."
}
```

Названия полей могут уточняться при переходе к технической реализации. Важно не конкретное имя каждого поля, а сама логика: после нормализации у системы должен быть структурированный объект, в котором отделены базовые параметры, жёсткие требования, мягкие предпочтения, уверенность интерпретации и текст preview.

### 4.5 Извлечение параметров и оценка уверенности

После получения пользовательского ввода система извлекает возможные параметры генерации и оценивает уверенность по каждому базовому параметру.

Оценка уверенности может использовать условную шкалу от 0 до 100.

Рабочие правила:

| Уверенность | Поведение системы                                                                                     |
| ----------- | ----------------------------------------------------------------------------------------------------- |
| `>= 75`     | Параметр принимается автоматически                                                                    |
| `50–74`     | Система предлагает наиболее вероятную интерпретацию и кратко объясняет, как она повлияет на результат |
| `< 50`      | Система считает параметр неясным и предлагает несколько подходящих вариантов                          |

Пример:

```text
Пользователь: Сделай про ёжика.
```

Возможная оценка:

| Параметр            | Значение           | Уверенность |
| ------------------- | ------------------ | ----------- |
| `content_format`    | `story`            | 80          |
| `main_subject`      | `ёжик`             | 95          |
| `truth_mode`        | неясно             | 30          |
| `utility_mode`      | неясно             | 30          |
| `target_age`        | неясно             | 10          |
| `output_count`      | дефолтное значение | 50          |
| `audience_language` | `ru`               | 95          |
| `result_language`   | `ru`               | 90          |

В таком случае система не должна задавать отдельный вопрос по каждому параметру. Вместо этого она должна сформировать пакетное уточнение с несколькими готовыми вариантами.

---

### 4.6 Пакетное уточнение параметров

Уточняющий диалог должен быть пакетным.

Оркестратор не должен задавать отдельный вопрос по каждому недостающему параметру:

```text
Какой возраст?
Какой режим?
Какой стиль?
Сколько историй?
```

Вместо этого он формирует одно компактное сообщение, в котором:

1. кратко сообщает, как понял запрос;
2. показывает, какие параметры неясны или могут быть выбраны по-разному;
3. предлагает один или несколько полных вариантов интерпретации;
4. даёт пользователю возможность выбрать вариант или уточнить своими словами.

Пример:

```text
Я понял, что вы хотите истории про ёжика, но можно сделать это по-разному.

Вариант 1 — правдивые короткие истории про ёжика в лесу для ребёнка 3 лет.
Вариант 2 — сказочная серия про маленького ёжика, который ищет друзей.
Вариант 3 — поучительные истории про ёжика, который учится делиться.

Можете выбрать вариант или написать свои уточнения.
```

Каждый предложенный вариант должен быть полным набором базовых параметров. Если пользователь выбирает вариант, система не должна после этого спрашивать базовые параметры, которые сама же могла заполнить в этом варианте.

Если пользователь отвечает свободным текстом, система снова анализирует ответ, обновляет параметры и проводит финальную проверку.

---

### 4.7 Лимит уточняющих итераций

Система не должна застревать в бесконечном уточняющем диалоге.

Максимальное количество уточняющих итераций — 5.

Если за это время не удалось получить все параметры явно, система мягко фиксирует наиболее вероятную интерпретацию и сообщает пользователю, как именно поняла запрос.

Пример:

```text
Я понял запрос примерно так: подготовить 3 короткие сказочные истории про ёжика для ребёнка 4 лет в спокойном повествовательном стиле.
```

После этого система показывает preview параметров и переходит к основному пайплайну, если финальный валидатор считает набор параметров внятным и непротиворечивым.

Исключение: если исходный ввод был пустым или бессмысленным и после лимита уточнений всё ещё не удалось получить внятную тему или выбранный пользователем вариант, система не должна автозапускать основной пайплайн. В этом случае приоритет имеет правило осмысленного пользовательского выбора: без темы, выбранного варианта или понятной интерпретации генерация не начинается.

---

### 4.8 Поведение при пустом или бессмысленном вводе

Если пользователь ничего не ввёл или ввёл бессмысленный текст, система не должна пытаться молча запускать генерацию.

Сначала она кратко объясняет, что умеет продукт:

```text
Я могу подготовить серию коротких детских историй: правдивых, сказочных, поучительных или обучающих.
```

Затем предлагает несколько готовых стартовых вариантов.

Пример:

```text
Можно попробовать:

1. 5 сказок про ёжика для ребёнка 4 лет.
2. 3 правдивые истории про белку зимой для ребёнка 3 лет.
3. 5 поучительных историй про маленького Мука, который учится быть смелым.
```

В отличие от обычного понятного запроса, при пустом или бессмысленном вводе система не должна начинать генерацию без выбора или подтверждения пользователя. Если система сама предложила исходный вариант, пользователь должен выбрать его или уточнить.

После выбора варианта система переходит к preview параметров.

Для пустого или бессмысленного ввода это правило имеет приоритет над общим лимитом уточнений: система может предложить варианты, но не должна запускать генерацию полностью на собственных догадках, если пользователь так и не выбрал вариант и не дал внятную тему.

---

### 4.9 Проверка согласованности параметров

Перед запуском основного пайплайна система должна проверить, что параметры не противоречат друг другу.

Противоречие может возникнуть между:

* режимом правдивости и сюжетной деталью;
* возрастом ребёнка и сложностью темы;
* utility-режимом и выбранным стилем;
* жёстким пользовательским требованием и правилами безопасности;
* выбранным подстилем и режимом генерации.

Пример противоречия:

```text
Расскажи правдивую поучительную историю про ёжика на ковре-самолёте.
```

Здесь конфликтуют:

```text
truth_mode = Правда
сюжетная деталь = ёжик летает на ковре-самолёте
```

Для режима `Правда` ковёр-самолёт как реальное событие не подходит. Система не должна молча игнорировать деталь или незаметно менять режим. Она должна явно сообщить о противоречии и предложить непротиворечивые варианты.

Пример ответа:

```text
В запросе есть противоречие: режим “Правда” предполагает реалистичную историю, а ковёр-самолёт — сказочный элемент.

Можно выбрать один из вариантов:

1. Оставить режим “Правда”: поучительная реалистичная история про ёжика в лесу.
2. Переключить в режим “Сказка”: история про ёжика, который летает на ковре-самолёте.
3. Сохранить правдивый режим, но перенести ковёр-самолёт в воображение ребёнка или рисунок.
```

Каждая альтернатива должна:

* быть внутренне непротиворечивой;
* сохранять максимум исходного намерения пользователя;
* содержать полный набор базовых параметров;
* быть исполнимой через доступные режимы и prompt layers.

---

### 4.10 Prompt-aware интерпретация запроса

Оркестратор не должен интерпретировать пользовательский запрос в отрыве от доступной базы режимов, стилей, подстилей и prompt layers.

Если система предлагает пользователю готовую интерпретацию запроса, эта интерпретация должна быть исполнимой: для неё должны существовать соответствующие режимы, описания, fallback-слои или prompt layers.

Пример:

```text
Хочу сказку как у Чуковского про ёжика.
```

Система должна сопоставить запрос с доступными параметрами:

```text
truth_mode = Сказка
utility_mode = Повествовательный
substyle = CHUKOVSKY
main_subject = ёжик
```

При этом агент не обязан использовать полный текст prompt layer в уточняющем диалоге. Для общения с пользователем достаточно краткого описания режима или стиля:

```text
Я понял так: это будет сказочная игровая история про ёжика, с ритмичной речью, повторами и лёгкой абсурдностью в духе детской поэзии.
```

Главный принцип:

```text
Система предлагает пользователю только те варианты, которые может исполнить через текущую prompt-базу.
```

---

### 4.11 Fallback-логика prompt layers

Если пользователь указывает стиль, подстиль, персонажа или другую деталь, для которой нет точного prompt layer, система не должна падать и не должна молча терять эту деталь.

Возможны три базовых сценария:

1. **Есть точный prompt layer**
   Система использует соответствующий слой.

2. **Точного prompt layer нет, но есть близкий поддерживаемый слой**
   Система может предложить пользователю ближайший поддерживаемый вариант.

3. **Нет точного или близкого слоя**
   Система использует базовый prompt layer, но сохраняет пользовательскую деталь как контекст — в `hard_details` или `soft_preferences`, в зависимости от силы требования.

Дополнительное правило: неподдерживаемый стиль или подстиль не всегда требует уточнения. Если пользователь сформулировал его как жёсткое требование, система должна уточнить или предложить поддерживаемую альтернативу. Если это мягкое пожелание, система может не останавливать сценарий, а сохранить пожелание в `soft_preferences` и продолжить на базовом или ближайшем поддерживаемом слое.

Пример:

```text
Пользователь: Сделай историю в стиле X про ёжика.
```

Если слоя `X` нет, система не должна отвечать так, будто стиль `X` полноценно поддерживается. Она должна либо предложить ближайшую поддерживаемую альтернативу, либо использовать базовый стиль и честно сохранить пожелание как дополнительную деталь.

Логика выбора:

```text
точный слой найден
        → использовать точный prompt layer

точный слой не найден, есть близкий слой
        → предложить близкий поддерживаемый вариант

точный слой не найден, близкого слоя нет
        → если требование жёсткое: уточнить или предложить альтернативу
        → если пожелание мягкое: использовать базовый prompt layer
        → сохранить исходную деталь в состоянии
        → не обещать пользователю точную стилизацию, которую система не поддерживает
```

Fallback-логика должна сохранять максимум исходного намерения пользователя, но не должна создавать ложное впечатление, что система поддерживает недоступный режим или стиль.

---

### 4.12 Самоописываемые prompt layers

Параметры генерации, режимы, стили и подстили будут развиваться. В системе будут появляться новые значения, новые режимы, новые возрастные профили, новые utility-режимы и новые prompt layers.

Поэтому каждый prompt layer, который система может использовать или предложить пользователю, должен быть самоописываемым.

Рекомендуемый формат — один `.md` файл, содержащий:

1. YAML metadata-блок;
2. основной prompt body.

Пример:

```markdown
---
id: CHUKOVSKY
name: Чуковский
type: substyle
truth_mode: FAIRY_TALE
user_description: Игровой, ритмичный, немного абсурдный стиль для детских сказочных историй.
short_description: Ритм, повторы, игра слов, динамика, лёгкая сказочная нелепость.
constraints:
  - Не копировать конкретные тексты.
  - Не делать лексику слишком сложной для возраста 3–5 лет.
  - Не перегружать текст абсурдом.
good_for:
  - сказочные истории
  - игровые истории
  - короткие ритмичные тексты
bad_for:
  - строгий режим правды
  - натуралистичные истории
  - энциклопедический стиль
examples:
  - Сказочная история про ёжика с повторами и весёлыми словами.
  - Игровая история с быстрым ритмом и простыми образами.
---

# Основной prompt layer

...
```

YAML metadata используется на этапе понимания и уточнения запроса. Основной markdown prompt body используется на этапе генерации, валидации или редактуры.

Такой подход позволяет:

* держать описание и prompt layer в одном месте;
* не допускать расхождения между тем, что система обещает пользователю, и тем, что реально делает генератор;
* валидировать prompt-базу автоматически;
* расширять систему новыми режимами и стилями без переписывания логики оркестратора;
* давать пользователю человеко-понятные объяснения выбранных параметров.

---

### 4.13 Финальная валидация параметров

Перед preview и запуском основного пайплайна финальный валидатор должен проверить, что набор параметров:

* полный;
* внятный;
* непротиворечивый;
* безопасный для детского контента;
* исполнимый с учётом доступных режимов и prompt layers.

За финальным валидатором остаётся последнее решение о переходе к основному пайплайну.

Даже если пользователь выбрал один из предложенных вариантов или уточнил запрос свободным текстом, система всё равно должна провести финальную проверку.

Упрощённая схема:

```text
Выбор / уточнение пользователя
        │
        ▼
Нормализация параметров
        │
        ▼
Финальная валидация параметров
        │
        ├─ pass → preview
        │
        └─ fail → пакетное уточнение / альтернативы / остановка
```

---

### 4.14 Preview параметров перед запуском генерации

После успешной финальной валидации система показывает пользователю friendly-preview параметров будущей генерации.

Preview не требует обязательного действия от пользователя. Его цель — дать пользователю коротко увидеть, как система поняла запрос, перед запуском основного пайплайна.

Пример:

```text
Я подготовлю серию историй:

Формат: истории
Режим: Правда
Тип: Повествовательные истории
Возраст: 3 года
Главный персонаж: ёжик
Место: зимний лес
Базовый стиль: спокойный
Подстиль: натуралистичный
Количество: 5 историй
Язык общения: русский
Язык результата: русский
```

Preview может отображаться короткое время, например 3–5 секунд, но это пример UX-реализации, а не логическое требование. В CLI, веб-интерфейсе, Telegram-боте или другом интерфейсе этот шаг может выглядеть по-разному.

В будущем может быть добавлена возможность отменить запуск или изменить параметры в течение preview-окна. На текущем уровне логики это считается дополнительной UX-возможностью, а не обязательным условием запуска.

Preview должен показываться всегда перед основным пайплайном, независимо от того, был ли исходный запрос полным, неполным, противоречивым или выбранным из предложенных вариантов.

### 4.16 Переход к основному пайплайну

После preview система переходит к основному пайплайну генерации.

К этому моменту должны быть выполнены условия:

1. базовые параметры заполнены;
2. жёсткие пользовательские требования зафиксированы;
3. противоречия устранены;
4. финальный валидатор разрешил переход;
5. пользователь увидел preview параметров.

Дальнейшая логика основного пайплайна включает:

* подбор prompt layers / `.md`-файлов под параметры;
* формирование плана серии;
* проверку и улучшение плана;
* генерацию текстов;
* проверку и улучшение текстов;
* возможное согласование с пользователем;
* генерацию иллюстраций;
* финальную выдачу результата.

Эти этапы описываются в следующих разделах документа.
