# 📋 ТЕХНИЧЕСКОЕ ЗАДАНИЕ НА РАЗРАБОТКУ DREAMYDRAW
## (для ИИ-агента-разработчика)

---

## Контекст проекта

DreamyDraw — это конвейер генерации детского контента: короткий текст → иллюстрация → вопросы для ребёнка. Пользователь задаёт тему (например, «сделай 3 иллюстрации про лису для ребёнка 3 лет»), выбирает режим правдивости, стили текста и картинки — и получает результат.

**Пользовательский гид (`WHAT_IS_DREAMYDRAW.md`) уже готов** и описывает, как система должна выглядеть и работать с точки зрения пользователя. Ваша задача — реализовать техническую часть, опираясь на этот гид, но не копируя его слепо.

---

## Общие требования к коду (критичны)

1. **SOLID** — каждый принцип должен быть соблюдён. Нарушения только с явным обоснованием в документации.

2. **Запрещены антипаттерны**:
   - God Object
   - Hardcoding (все настройки — через конфиги, фабрики, реестры)
   - Singleton (там, где он не нужен)
   - Spaghetti code
   - Copy-paste

3. **Расширяемость без изменения ядра**:
   - Новый провайдер (LLM или генератор картинок) = новый файл + регистрация. Ядро не трогать.
   - Новый режим правдивости = новый файл. Ядро не трогать.
   - Новый стиль текста / рисунка = новый файл. Ядро не трогать.
   - Новый «мир» (животные, город, мифы) = новая конфигурация или новый файл — на ваше усмотрение, но без правки оркестратора.

4. **Базы данных не используются**. Всё хранение — JSON-файлы (кеш, состояние сессий, метрики использованных токенов/генераций). Вы предложите структуру в документации.

5. **Работа с состоянием**: система должна уметь восстанавливаться после прерывания (например, сохранять промежуточные шаги при `mode=check`). Кеширование идентичных запросов — обязательно.

## Этапы разработки (после каждого — согласование с руководителем)

### Этап 1. Документация

Подготовить в папке `/docs` следующие документы (формат Markdown):

#### 1.1 `ARCHITECTURE.md`
- Общая архитектура (компоненты, их ответственность, связи)
- Диаграмма потока данных (можно текстовой схемой)
- Принципы расширения (как добавить нового провайдера, режим, стиль, мир)
- Почему выбрана такая архитектура (альтернативы рассмотрены)

#### 1.2 `MODULES.md`
- Список модулей с их интерфейсами (не реализация, а контракты)
- Каждый модуль: что делает, от чего зависит, кто его вызывает
- Где лежит конфигурация

#### 1.3 `REQUIREMENTS.md`
Специфические требования, не из SOLID:
- Два режима работы: `fast` (сразу текст+картинка) и `check` (сначала текст, после подтверждения пользователя — картинка)
- Пользователь задаёт количество иллюстраций естественным языком («сделай 3 иллюстрации», «сделай 5 картинок»). Если не указано — по умолчанию 3.
- На каждом шаге (история) генерируются: текст (3–5 предложений), иллюстрация с текстом внутри (текст накладывается через промпт, не постобработкой), вопросы (2–3 шт.)
- Режимы правдивости: «Правда», «Миф», «Сказка» (три дискретных варианта)
- Стили текста: перечислены в гиде пользователя (минимум 3)
- Стили рисунка: перечислены в гиде пользователя (минимум 4)
- Кеш: одинаковые запросы (тема + режим + стили) не генерируются заново. Реализацию предлагаете вы.
- Хранение: JSON (никаких БД)
- Текст внутри картинки генерируется промптом к генератору изображений. Не постобработка.

#### 1.4 `ROADMAP.md`
Что в каком порядке реализуется (см. этапы 2–9 ниже). Отметить, что будет готово на этапе 2, что позже.

Дополнительные документы — на усмотрение разработчика.

### Этап 2. Прототип с рабочей архитектурой и обрезанным функционалом

**Цель:** получить масштабируемое ядро, которое можно будет наращивать без переписывания.

**Что обязательно реализовать:**

- Запуск через терминал (CLI). Пользователь указывает:
  - тему (строка, обязательный параметр)
  - режим правдивости (`truth_mode`)
  - стиль текста (`text_style`)
  - стиль картинки (`image_style`)
  - режим работы (`fast` или `check`)
  - количество иллюстраций (опционально, алгоритм может сам извлечь из текста темы либо использовать значение по умолчанию 3)

- Оркестратор и пайплайн (без LangGraph / LangFuse). Реализуете сами, с полным контролем. Шаги пайплайна:
  1. Парсинг входа (извлечь количество иллюстраций из темы)
  2. Генерация текстов (для каждой иллюстрации) — пока мок
  3. Если `mode=check` — остановка, запрос подтверждения, возможность перегенерации текста
  4. Генерация картинок (с текстом внутри, через промпт) — пока мок
  5. Генерация вопросов — пока мок
  6. Сохранение результата (JSON + отдельные файлы для текстов/картинок)

- Все моки должны быть реализованы как настоящие провайдеры (через фабрику/реестр), чтобы замена на реальный API была точечной.

- Кеширование: при повторном запуске с теми же параметрами — берётся из кеша.

- JSON-хранилище для кеша и промежуточных состояний.

**Чего НЕ надо делать на этапе 2:**
- Веб-интерфейс
- Реальные провайдеры (только моки)
- Сложные промпты (достаточно шаблонов вида: «напиши рассказ про {topic} в режиме {mode} для ребёнка 3 лет»)
- Авторизацию, оплату, личные кабинеты

**После этапа 2 — демонстрация руководителю:** запуск CLI с разными параметрами, вывод результатов (текст, картинка-заглушка, вопросы), проверка работы кеша и режима `check`.

### Этап 3. Доработка промптов

На этом этапе разработчик готовит промпты в простейшем варианте (логика работы уже есть) и предоставляет их руководителю на корректировку. Руководитель вносит правки в промпты (или даёт задания на доработку). Разработчик не меняет ядро, только содержимое промптов.

---

### Этап 4. Подключение реальных провайдеров

Разработчик подключает реальные API:
- LLM (через GPTunnel или аналоги — архитектура должна позволять подключать любого провайдера без изменения ядра)
- Генератор изображений (Bananapi / DALL-E / Stable Diffusion — через тот же агрегатор или отдельно)

Требование: возможность в будущем добавлять новых провайдеров (другой LLM или другого генератора картинок) путём создания нового файла и регистрации в реестре. Ядро не трогать.

---

### Этап 5. Эксперименты с оркестратором

Разработчик пробует разные варианты организации оркестрации (в рамках уже существующего ядра, не ломая пайплайн). Документирует, что получилось, какие боли возникли. Это нужно для осознанного перехода на LangGraph/LangFuse.

---

### Этап 6. Переход на LangGraph и LangFuse

Оркестратор переписывается с использованием LangGraph. Пайплайн и бизнес-логика сохраняются. LangFuse добавляется для observability.

---

### Этап 7. Веб-приложение (локальная версия)

**Ограничения:**
- Не использовать чистый HTML (можно фреймворки, шаблонизаторы, SPA — на усмотрение разработчика)
- Веб-сервер должен подниматься локально (по хосту и порту, которые можно сконфигурировать)
- Должна быть заложена возможность безболезненного переезда на удалённую машину (конфигурация через переменные окружения)
- Должна быть заложена возможность смены домена (относительные пути, настройки CORS и т.д.)
- Архитектура фронтенда должна позволить в будущем добавить авторизацию и личный кабинет (т.е. не завязываться на отсутствие аутентификации)

**Функционал веб-версии (полностью соответствует пользовательскому гиду):**
- Поля для ввода темы (свободный текст)
- Дискретные кнопки/выборщики для:
  - Режимов правдивости (3 варианта)
  - Стилей текста (3+ варианта)
  - Стилей картинки (4+ варианта)
- Выбор режима работы (fast / check)
- Возможность указать количество иллюстраций (или распарсить из темы — на ваше усмотрение)
- Отображение полученных историй (текст + картинка + вопросы), с возможностью скрыть картинку (как указано в гиде)
- В режиме check — интерфейс подтверждения каждого текста перед генерацией картинки

### Этап 8. Docker

Разработчик упаковывает всё в Docker:
- Dockerfile (многоступенчатый, если нужно)
- docker-compose.yml для локальной разработки и запуска
- Переменные окружения вынесены из кода

---

### Этап 9. Авторизация, оплата, личный кабинет

**Этот этап — отдельный, сложный. Разработчик должен предложить архитектуру, затем реализовать.**

Функционал:
- Авторизация по email (магическая ссылка или пароль — на усмотрение разработчика)
- После регистрации — пробный период: не более 10 иллюстраций (рассказов + картинок) бесплатно
- Система баланса:
  - Пользователь вносит деньги (интеграция с платёжным шлюзом — выбор разработчика с обоснованием)
  - Учёт потраченных средств за каждую генерацию (разные провайдеры могут стоить по-разному)
  - Отображение в личном кабинете: внесено / потрачено / остаток
- Безопасность: данные пользователей не должны храниться в открытом виде (хэши паролей, токены). Баланс — JSON, но с учётом целостности.

**Важно:** архитектура этапов 7–8 должна позволить добавить эту функциональность без переписывания ядра генерации.

1. Начинаете с **Этапа 1** (документация). После завершения — отправляете мне (руководителю) на согласование.
