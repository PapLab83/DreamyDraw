# ARCHITECTURE.md - DreamyDraw

## 1. Общая архитектура
Система строится по принципу **слоистой архитектуры (Layered Architecture)** с использованием **Dependency Injection** для обеспечения расширяемости.

### Основные компоненты:
1.  **Entry Points (CLI/Web):** Взаимодействие с пользователем, парсинг аргументов.
2.  **Orchestrator (Core):** Управляет жизненным циклом генерации, координирует вызовы провайдеров, обрабатывает переходы состояний (особенно в режиме `check`).
3.  **Domain Models (Pydantic):** Типизированные структуры данных для историй, сессий и конфигураций.
4.  **Providers (Adapters):** Интерфейсы для работы с LLM и Image Generation API. Реализации скрыты за абстракциями.
5.  **Storage Layer:** Работа с JSON-файлами. Включает механизмы кеширования и управления сессиями.

## 2. Диаграмма потока данных (Data Flow)
```text
User -> CLI/Web -> Orchestrator -> [Provider Registry]
                                        |
    [Storage (JSON)] <--- Orchestrator <--> [LLM Provider] (Генерация текста/вопросов)
            |                           |
    [Storage (Cache)] <----------------> [Image Provider] (Генерация картинки)
            |
            V
       Final Output (JSON + JPG)
```

## 3. Принципы расширения
-   **Новый провайдер (LLM/Image):** Реализовать интерфейс `BaseLLMProvider` или `BaseImageProvider` в новом файле и зарегистрировать в `ProviderFactory`.
-   **Новый режим правдивости / Стиль:** Добавляется как элемент `Enum` или запись в реестре конфигураций. Логика промптов вынесена в шаблоны, которые выбираются на основе этих параметров.
-   **План Б (Постобработка):** В пайплайне предусмотрен опциональный шаг `ImageEnhancer`, который может накладывать текст на изображение с помощью PIL, если флаг `use_postprocessing` активен.

## 4. Механизмы надежности
-   **File Locking:** Для предотвращения повреждения JSON-файлов при конкурентном доступе будет использоваться библиотека `portalocker` или аналогичный механизм блокировок на уровне ОС.
-   **Session State:** Каждая генерация имеет уникальный `session_id`. Состояние сохраняется после каждого шага (текст сгенерирован -> сохранение -> ожидание подтверждения -> генерация картинки).

## 5. Почему такая архитектура?
-   **SOLID:** Позволяет менять реализацию генерации картинок (например, с DALL-E на Stable Diffusion) без изменения логики оркестратора (Open-Closed Principle).
-   **Масштабируемость:** Разделение на модули упрощает переход на LangGraph в будущем (Этап 6).
-   **Альтернативы:** Рассматривалась архитектура монолитного скрипта, но она была отвергнута из-за нарушения требований к расширяемости и сложности тестирования.
